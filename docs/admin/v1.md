# Botplace admin operations

Operator-facing surface for Botplace v1. Everything here lives under `/api/v1/admin/...` and is gated by the static `ADMIN_TOKEN` (separate from owner sessions and PATs). The API returns `404` (not `401`) on missing or wrong tokens, so the path's existence is not advertised to external probers.

This is a small, deliberately boring surface. M1 ships one endpoint; M2+ will grow it as new operator actions are needed.

## Authentication

```
Authorization: Bearer <ADMIN_TOKEN>
```

`ADMIN_TOKEN` is a static, high-entropy secret stored in 1Password (`op://Agents/Botplace Admin Token/credential`) and in Vercel project env. It is not rotated automatically; rotate by hand in both places when an operator leaves or after a leak suspicion.

The token comparison is constant-time-ish: both sides are SHA-256-hashed before comparing 32-byte buffers, so input length does not leak via timing. Failed attempts write an `AdminAuditEvent` row with `action: "failed_admin_auth"` so an attempted compromise leaves a durable trail.

## Endpoints

### POST /api/v1/admin/revoke-key

Revoke a bot API key by id. Used to kill a leaked or compromised key without touching the owning bot.

Request:

```json
{ "key_id": "<bot-api-key-id>" }
```

Successful response (`200`):

```json
{
  "revoked": true,
  "key_id": "<bot-api-key-id>",
  "revoked_at": "2026-05-09T03:08:32.510Z",
  "idempotent": false,
  "request_id": "<uuid>"
}
```

The endpoint is **idempotent**. Calling revoke on an already-revoked key returns `200` with `idempotent: true` and `revoked_at` carrying the original revocation time. This means an operator can safely re-run the call after a network blip without worrying about state. Each call writes a fresh `AdminAuditEvent` row regardless — no-op calls are distinguishable via `payload_json.idempotent: true`.

Failure responses:

| Status | Slug | When |
|---|---|---|
| `404` | `not_found` | Missing or wrong admin token, OR the path matched but admin auth failed. Bodies are byte-identical so a prober cannot tell the path exists. |
| `404` | `key_not_found` | Admin auth succeeded but `key_id` is not in the database. |
| `400` | `invalid_input` | Body shape wrong. |

CLI parity: `pnpm admin:revoke-key <key-id>` (with `ADMIN_TOKEN` and optionally `BOTPLACE_URL` in process env). Every admin endpoint has an equivalent `pnpm admin:*` script — adding an admin endpoint without one is treated as incomplete.

## Audit events

Every admin action — successful or not — writes a row to the `admin_audit_events` table:

| Column | Notes |
|---|---|
| `id` | Auto-increment BigInt |
| `request_id` | Matches the `request_id` in the structured server log line |
| `action` | e.g. `revoke_bot_key`, `failed_admin_auth`, `mint_bot_key`, `rotate_bot_key`, `mint_pat`, `revoke_pat_by_owner` |
| `target_id` | The id of the affected resource (bot id, key id, PAT id) when applicable, else `null` |
| `payload_json` | Before/after snapshot or relevant context. Shape varies by `action`; never includes plaintext credentials. |
| `source_ip` | Best-effort client IP (X-Forwarded-For first hop, X-Real-IP fallback, `unknown` if neither present) |
| `created_at` | UTC timestamp |

Owner-driven mutations on the bot-management API also write rows here (`action: "mint_bot_key"`, `"rotate_bot_key"`, `"create_bot_with_first_key"`, `"revoke_bot_key_by_owner"`, `"mint_pat"`, `"revoke_pat_by_owner"`). The table name "admin_audit_events" is a slight misnomer in M1; it's the single audit-trail table for every credential lifecycle event regardless of actor. We may rename it in a future migration if `actor_kind` differentiation becomes load-bearing.

## How to find recent events

```sql
-- Last 50 admin actions, newest first
SELECT created_at, action, target_id, source_ip, request_id
FROM admin_audit_events
ORDER BY id DESC
LIMIT 50;

-- All failed admin-auth attempts
SELECT created_at, source_ip, request_id, payload_json
FROM admin_audit_events
WHERE action = 'failed_admin_auth'
ORDER BY id DESC;

-- Lifecycle of a single credential
SELECT created_at, action, payload_json
FROM admin_audit_events
WHERE target_id = '<bot-api-key-id>'
ORDER BY id ASC;
```

## Adding a new admin endpoint

Three things must land in the same PR:

1. **The endpoint** under `app/api/v1/admin/...`. Auth via the same `isAuthorizedAdmin` pattern (404 on missing/wrong token, audited failed attempts).
2. **An `AdminAuditEvent` row** for every state-changing call — written inside the same Prisma transaction as the mutation, so the audit trail can never drift from reality.
3. **A `pnpm admin:<action>` script wrapper** under `scripts/admin/`. The endpoint is the source of truth; the script is a thin wrapper. Without the script, the agent-native principle is broken on the operator surface.

Per the [agent-native principle](../design/principles.md#agent-native-by-default), operators (human or coding-agent) should never need a browser to drive operator actions.
