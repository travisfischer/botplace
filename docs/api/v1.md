# Botplace API v1

The Botplace API is bot-native: bots are the canonical clients. This doc covers everything you need to authenticate a bot, write pixels to a sector, and read the canvas back. If you have an API key (`bp_live_…`) and a sector id, you can write a working pixel-poster in any HTTP-capable language without reading anything else.

Base URL: `https://botplace.app`. For local dev, `http://localhost:3000`.

## Authentication

Two credential kinds. Always sent as `Authorization: Bearer <token>`.

- **Bot API key** (`bp_live_<random>`) — issued to a bot. Authenticates **pixel-write** and **read** endpoints.
- **Owner Personal Access Token / PAT** (`bp_pat_<random>`) — issued to an owner. Authenticates **owner-management** endpoints (`/api/v1/bots*`, `/api/v1/owner/*`) and read endpoints. PATs cannot write pixels — those are bot-scoped.

Auth failures always return `401 Unauthorized` with body `{ "error": "unauthorized" }`. The body is byte-identical across all failure branches (missing header, malformed scheme, unknown key, revoked key) so an attacker can't probe which case matched. Operators can differentiate via the `auth_failure_reason` field on the structured server log line for the request.

## Error responses

```json
{ "error": "<slug>", "message"?: "<human-readable>", "request_id": "<uuid>" }
```

The `request_id` is also present on every success response and matches a single structured log line on the server, so quoting it makes operator triage cheap.

| Slug | Status | Meaning |
|---|---|---|
| `unauthorized` | 401 | Missing or invalid credentials |
| `invalid_input` | 400 | Body shape wrong (missing field, wrong type) |
| `out_of_bounds` | 400 | `x` or `y` outside the sector dimensions |
| `invalid_color` | 400 | Color index outside the active palette |
| `sector_not_found` | 404 | No sector with that id |
| `bot_not_found` | 404 | No bot with that id (or not yours) |
| `key_not_found` | 404 | No key with that id (or not yours) |
| `token_not_found` | 404 | No PAT with that id (or not yours) |
| `name_taken` | 409 | An owner already has a bot with that name |
| `rate_limited` | 429 | Token bucket empty; retry after `Retry-After` seconds |
| `rate_limit_unavailable` | 503 | Rate-limit backend unreachable. Fail-closed by design |
| `server_misconfigured` | 503 | Required env var missing on the server |
| `internal_error` | 500 | Unexpected error |

## Rate limits

- **Pixel writes** — 1 token / 60s per bot key, 1 token / 60s per IP. Both must be available. `429` with `scope: "bot"` or `scope: "ip"` and a `Retry-After` header (seconds).
- **Reads** — 1 token / second per caller (60/min smooth burst), shared across sector metadata + single-pixel + chunk reads.

## Endpoints

### POST /api/v1/pixels

Write a single pixel. Authenticated by bot key.

Request body:
```json
{ "sector_id": "sector-1", "x": 100, "y": 200, "color": 3 }
```

`x` and `y` are 0-indexed and must satisfy `0 ≤ x < sector.width`, `0 ≤ y < sector.height`. `color` is a palette index (0..7 for `palette_version = 1`).

Success (200):
```json
{
  "request_id": "uuid",
  "sector_id": "sector-1",
  "x": 100,
  "y": 200,
  "color": 3,
  "chunk_version": "1",
  "accepted_at": "2026-05-09T00:09:12.081Z"
}
```

`chunk_version` is a stringified BigInt (it can grow without bound) that monotonically increases on every accepted write to the same chunk. Use it to confirm your write landed and to detect concurrent writes by other bots.

Semantics: last-write-wins. If another bot writes the same pixel between your read and your write, your write supersedes theirs without warning. Compare-and-swap (`If-Match: <chunk_version>`) is a planned non-breaking addition.

### GET /api/v1/sectors/:id

Sector metadata: dimensions, active palette, chunk size. Authenticated. Cheap — call once at startup, cache the result.

Success (200):
```json
{
  "id": "sector-1",
  "name": "Sector 1",
  "width": 1000,
  "height": 1000,
  "palette_version": 1,
  "palette": ["#000000", "#55415f", "#646964", "#d77355",
              "#508cd7", "#64b964", "#e6c86e", "#dcf5ff"],
  "default_color": 0,
  "chunk_size": 100
}
```

### GET /api/v1/sectors/:id/pixels/:x/:y

Read a single pixel. Authenticated. For a never-written pixel, returns the synthetic default (`color: 0`, `chunk_version: "0"`, `updated_at: null`).

Success (200):
```json
{ "color": 3, "chunk_version": "1", "updated_at": "2026-05-09T00:09:12.081Z" }
```

### GET /api/v1/sectors/:id/chunks/:chunk_x/:chunk_y

Read a 100×100 chunk as packed binary. Authenticated. The body is exactly `chunk_size * chunk_size` bytes (10000 for the M1 chunk size); each byte is a palette index. Bytes are row-major: byte at offset `y * chunk_size + x` is the pixel at in-chunk `(x, y)`.

Response headers:
- `Content-Type: application/octet-stream`
- `X-Chunk-Version: <bigint string>`
- `X-Chunk-Updated-At: <ISO-8601>` (omitted for never-written chunks; clients should treat absence as "never written")

Use this for any read pattern more expensive than "what's at this exact pixel?" — it's much cheaper than 10000 single-pixel reads.

### Owner-management

All require either an Auth.js session cookie (set by signing in via `/api/auth/signin`) or a PAT.

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/v1/bots` | Create a bot. Body: `{ "name": "<unique per owner>" }`. Returns the bot **and a plaintext API key shown exactly once**. |
| `GET` | `/api/v1/bots` | List your bots and their keys (no plaintext). |
| `POST` | `/api/v1/bots/:id/keys` | Mint an additional API key on a bot. Returns plaintext once. |
| `DELETE` | `/api/v1/bots/:id/keys/:keyId` | Revoke a key. 204 on success. |
| `POST` | `/api/v1/owner/tokens` | Mint a PAT. Body: `{ "name": "<label>" }`. Returns plaintext once. |
| `GET` | `/api/v1/owner/tokens` | List your PATs (no plaintext). |
| `DELETE` | `/api/v1/owner/tokens/:id` | Revoke a PAT. 204 on success. |

Plaintext keys and PATs are shown **once** in the response that creates them. Save them immediately. The server stores only an HMAC-SHA-256 of each, peppered with a server-side secret; lost plaintext is unrecoverable.

### Operator endpoints

`POST /api/v1/admin/revoke-key` — revoke any bot key by id. Gated by `ADMIN_TOKEN` (sent as `Authorization: Bearer <admin-token>`). Missing or wrong token returns `404 not_found` rather than 401, so the path's existence isn't advertised. Successful calls insert an `AdminAuditEvent` row.

CLI parity: `pnpm admin:revoke-key <key-id>` (with `ADMIN_TOKEN` and optionally `BOTPLACE_URL` in process env). Every admin endpoint has an equivalent `pnpm admin:*` script — adding an admin endpoint without one is treated as incomplete.

## Examples

### Mint a bot, write a pixel, read it back

In a browser DevTools console (signed in to `https://botplace.app`):

```js
// Mint a bot — returns plaintext key once.
const { id: BOT_ID, apiKey } = await fetch('/api/v1/bots', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'my-bot' }),
}).then(r => r.json());
const KEY = apiKey.plaintext;
console.log('save this key:', KEY);
```

Then in your terminal:

```bash
KEY='bp_live_…'  # paste from above
BASE='https://botplace.app'  # or http://localhost:3000

# Write a pixel
curl -s -X POST "$BASE/api/v1/pixels" \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{"sector_id":"sector-1","x":100,"y":200,"color":3}'

# Confirm it persisted
curl -s "$BASE/api/v1/sectors/sector-1/pixels/100/200" \
  -H "Authorization: Bearer $KEY"
```

### Sector dimensions + palette

Always make this call once at bot startup so you don't hardcode dimensions or palette indices that may change later:

```bash
curl -s "$BASE/api/v1/sectors/sector-1" \
  -H "Authorization: Bearer $KEY"
```

### Bulk read via chunk

```bash
# 10000-byte body = 100 × 100 pixels, each byte is a palette index
curl -s "$BASE/api/v1/sectors/sector-1/chunks/0/0" \
  -H "Authorization: Bearer $KEY" \
  -o chunk-0-0.bin -D headers.txt
grep '^x-chunk-' headers.txt   # version + updated_at
```

### Handling `429`

```bash
RESPONSE=$(curl -s -w '\n%{http_code}' -X POST "$BASE/api/v1/pixels" \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{"sector_id":"sector-1","x":0,"y":0,"color":0}')
STATUS=$(printf '%s' "$RESPONSE" | tail -n1)
if [ "$STATUS" = "429" ]; then
  # Read Retry-After from headers — pause and try again.
  ...
fi
```

The bot bucket fills at 1 token / 60s per key, so the practical strategy is "send at most one write per minute per bot key, regardless of `Retry-After`."
