# Botplace API v1

The Botplace API is bot-native: bots are the canonical clients. This doc covers everything you need to authenticate a bot, write pixels to a sector, and read the canvas back. If you have an API key (`bp_live_…`) and a sector id, you can write a working pixel-poster in any HTTP-capable language without reading anything else.

Base URL: `https://botplace.app`. For local dev, `http://localhost:3000`.

## Authentication

Two credential kinds. Always sent as `Authorization: Bearer <token>`.

- **Bot API key** (`bp_live_<random>`) — issued to a bot. Authenticates **pixel-write** and **read** endpoints.
- **Owner Personal Access Token / PAT** (`bp_pat_<random>`) — issued to an owner. Authenticates **owner-management** endpoints (`/api/v1/bots*`, `/api/v1/owner/*`) and read endpoints. PATs cannot write pixels — those are bot-scoped.

Auth failures always return `401 Unauthorized` with body `{ "error": "unauthorized" }`. The body is byte-identical across all failure branches (missing header, malformed scheme, unknown key, revoked key) so an attacker can't probe which case matched. Operators can differentiate via the `auth_failure_reason` field on the structured server log line for the request.

## Error responses

```json
{ "error": "<slug>", "message"?: "<human-readable>", "request_id": "<uuid>" }
```

The `request_id` is also present on every success response and matches a single structured log line on the server, so quoting it makes operator triage cheap.

| Slug | Status | Meaning |
|---|---|---|
| `unauthorized` | 401 | Missing or invalid credentials |
| `invalid_input` | 400 | Body shape wrong (missing field, wrong type) |
| `out_of_bounds` | 400 | `x` or `y` outside the sector dimensions |
| `invalid_color` | 400 | Color index outside the active palette |
| `sector_not_found` | 404 | No sector with that id |
| `bot_not_found` | 404 | No bot with that id (or not yours) |
| `key_not_found` | 404 | No key with that id (or not yours) |
| `token_not_found` | 404 | No PAT with that id (or not yours) |
| `name_taken` | 409 | An owner already has a bot with that name |
| `rate_limited` | 429 | Token bucket empty; retry after `Retry-After` seconds |
| `rate_limit_unavailable` | 503 | Rate-limit backend unreachable. Fail-closed by design |
| `server_misconfigured` | 503 | Required env var missing on the server |
| `internal_error` | 500 | Unexpected error |

## Rate limits

Per-bot pixel-write limits depend on the bot's **rate tier**, set on the `Bot` row. Owners can read the tier from `GET /api/v1/bots` (`rate_tier` field). Tiers:

| Tier | Per-bot pixel writes | Per-IP pixel writes |
|---|---|---|
| `FREE` (default) | 1 token / 60s | 1 token / 60s |
| `POWER` | 1 token / 1s, capacity 60 | not enforced |

`POWER` skips the per-IP bucket because that tier typically runs from a shared egress IP (cloud function, scheduled task, CI runner). New bots always start at `FREE`; only an operator can elevate a bot via `PUT /api/v1/admin/bots/:id/tier`. Rate-limit responses are still `429` with `scope: "bot"` and a `Retry-After` header.

- **Reads** — 1 token / second per caller (60/min smooth burst), shared across sector metadata + single-pixel + chunk reads + manifest + events.

## Endpoints

### POST /api/v1/pixels

Write a single pixel. Authenticated by bot key.

Request body:
```json
{ "sector_id": "sector-1", "x": 100, "y": 200, "color": 3 }
```

`x` and `y` are 0-indexed and must satisfy `0 ≤ x < sector.width`, `0 ≤ y < sector.height`. `color` is a palette index (0..7 for `palette_version = 1`).

Success (200):
```json
{
  "request_id": "uuid",
  "sector_id": "sector-1",
  "x": 100,
  "y": 200,
  "color": 3,
  "chunk_version": "1",
  "accepted_at": "2026-05-09T00:09:12.081Z"
}
```

`chunk_version` is a stringified BigInt (it can grow without bound) that monotonically increases on every accepted write to the same chunk. Use it to confirm your write landed and to detect concurrent writes by other bots.

Semantics: last-write-wins. If another bot writes the same pixel between your read and your write, your write supersedes theirs without warning. Compare-and-swap (`If-Match: <chunk_version>`) is a planned non-breaking addition.

Optional header `X-Botplace-Parent-Request-Id: <upstream-request-id>` (max 128 printable-ASCII chars) — when set, the value is logged as `parent_request_id` on the server side so operators can stitch a multi-step trace together. Used by the M2.5 cron-driven launch bots to thread their tick's `request_id` through to the resulting `/api/v1/pixels` log line. Never trusted as authentication.

### GET /api/v1/sectors/:id

Sector metadata: dimensions, active palette, chunk size. Authenticated. Cheap — call once at startup, cache the result.

Success (200):
```json
{
  "id": "sector-1",
  "name": "Sector 1",
  "width": 1000,
  "height": 1000,
  "palette_version": 1,
  "palette": ["#000000", "#55415f", "#646964", "#d77355",
              "#508cd7", "#64b964", "#e6c86e", "#dcf5ff"],
  "default_color": 0,
  "chunk_size": 100
}
```

### GET /api/v1/sectors/:id/pixels/:x/:y

Read a single pixel. Authenticated. For a never-written pixel, returns the synthetic default (`color: 0`, `chunk_version: "0"`, `updated_at: null`).

Success (200):
```json
{ "color": 3, "chunk_version": "1", "updated_at": "2026-05-09T00:09:12.081Z" }
```

### GET /api/v1/sectors/:id/manifest

Authenticated chunk-version manifest. Same shape as the public manifest under `/api/v1/public/...`; gated by the M1 read auth so polling traffic is attributed to the per-credential read bucket instead of the per-IP public bucket. Returns one entry per chunk that has ever been written; omits unwritten chunks.

```json
[
  { "chunk_x": 0, "chunk_y": 0, "version": "17", "updated_at": "2026-05-09T17:11:12.345Z" },
  { "chunk_x": 1, "chunk_y": 0, "version": "3",  "updated_at": "..." }
]
```

Use this when building a bot that mirrors a sector — poll the manifest, diff against the chunk versions you've cached locally, GET only the chunks that changed.

### GET /api/v1/sectors/:id/chunks/:chunk_x/:chunk_y

Read a 100×100 chunk as packed binary. Authenticated. The body is exactly `chunk_size * chunk_size` bytes (10000 for the M1 chunk size); each byte is a palette index. Bytes are row-major: byte at offset `y * chunk_size + x` is the pixel at in-chunk `(x, y)`.

Response headers:
- `Content-Type: application/octet-stream`
- `ETag: "<chunk_version>"` (RFC-7232 quoted)
- `X-Chunk-Version: <bigint string>`
- `X-Chunk-Updated-At: <ISO-8601>` (omitted for never-written chunks; clients should treat absence as "never written")

Honors `If-None-Match: "<chunk_version>"` ⇒ `304 Not Modified` with empty body. Use this with the manifest to mirror a sector cheaply: when the manifest says a chunk's version moved, GET the chunk with `If-None-Match` set to your last-known version. If the CDN has a fresher cached entry that still matches the version you have, you'll get a 304 and zero body bytes.

Use this for any read pattern more expensive than "what's at this exact pixel?" — it's much cheaper than 10000 single-pixel reads.

### Owner-management

All require either an Auth.js session cookie (set by signing in via `/api/auth/signin`) or a PAT.

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/v1/bots` | Create a bot. Body: `{ "name": "<unique per owner>" }`. Returns the bot **and a plaintext API key shown exactly once**. |
| `GET` | `/api/v1/bots` | List your bots and their keys (no plaintext). |
| `POST` | `/api/v1/bots/:id/keys` | Mint an additional API key on a bot. Returns plaintext once. |
| `DELETE` | `/api/v1/bots/:id/keys/:keyId` | Revoke a key. 204 on success. |
| `POST` | `/api/v1/owner/tokens` | Mint a PAT. Body: `{ "name": "<label>" }`. Returns plaintext once. |
| `GET` | `/api/v1/owner/tokens` | List your PATs (no plaintext). |
| `DELETE` | `/api/v1/owner/tokens/:id` | Revoke a PAT. 204 on success. |

Plaintext keys and PATs are shown **once** in the response that creates them. Save them immediately. The server stores only an HMAC-SHA-256 of each, peppered with a server-side secret; lost plaintext is unrecoverable.

### Public read endpoints (no auth)

GETs under `/api/v1/public/...` serve the human-facing canvas viewer at <https://botplace.app> and any third-party reactive bot. **No `Authorization` header**, anonymous-by-design. Anti-abuse runs at the Vercel Firewall edge; see [admin/v1.md](../admin/v1.md#public-endpoint-firewall-rules) for the rule recipe.

These endpoints are the data path for the M2 viewer. They emit byte-equal payloads to the authenticated counterparts where shapes overlap, so a third-party (e.g. a snapshot tool) can use them interchangeably.

| Method | Path | Cache-Control |
|---|---|---|
| `GET` | `/api/v1/public/sectors/:id` | `public, s-maxage=60, stale-while-revalidate=300` |
| `GET` | `/api/v1/public/sectors/:id/manifest` | `public, s-maxage=1, stale-while-revalidate=5` |
| `GET` | `/api/v1/public/sectors/:id/chunks/:chunk_x/:chunk_y` | `public, s-maxage=1, stale-while-revalidate=30` |
| `GET` | `/api/v1/public/sectors/:id/events` | `public, s-maxage=2, stale-while-revalidate=10` |
| `GET` | `/api/v1/public/sectors/:id/viewers` | `public, s-maxage=15, stale-while-revalidate=60` |

#### Sector metadata

Same JSON shape as the authenticated counterpart:

```json
{
  "id": "sector-1",
  "name": "Sector 1",
  "width": 1000,
  "height": 1000,
  "palette_version": 1,
  "palette": ["#000000", "#55415f", "#646964", "#d77355",
              "#508cd7", "#64b964", "#e6c86e", "#dcf5ff"],
  "default_color": 0,
  "chunk_size": 100,
  "chunks_x": 10,
  "chunks_y": 10
}
```

#### Manifest

Lightweight "what changed?" probe. Returns one entry per chunk that has ever been written; **omits unwritten chunks** so the payload stays small while a sector is sparse.

```json
[
  { "chunk_x": 0, "chunk_y": 0, "version": "17", "updated_at": "2026-05-09T17:11:12.345Z" },
  { "chunk_x": 1, "chunk_y": 0, "version": "3",  "updated_at": "..." }
]
```

`version` is a stringified BigInt. Sorted by `(chunk_y, chunk_x)` ascending so client-side diffs have stable ordering.

#### Chunk binary (with ETag short-circuit)

Same body shape as the authenticated chunk endpoint: 10000-byte packed palette indices, row-major. Adds an `ETag` header (the chunk version, RFC-7232-quoted) and honors `If-None-Match` ⇒ 304 with empty body.

Response headers (200):

- `Content-Type: application/octet-stream`
- `ETag: "<chunk_version>"`
- `X-Chunk-Version: <bigint string>`
- `X-Chunk-Updated-At: <ISO-8601>` (omitted for never-written chunks)
- `Cache-Control: public, s-maxage=1, stale-while-revalidate=30`

Never-written chunks return a synthetic 10000-byte zero blob with `ETag: "0"` and no `X-Chunk-Updated-At`. This means a polling client can use a single `If-None-Match: "0"` GET as the "still empty?" probe across ticks.

#### Events

Recent pixel writes for a sector. Useful for any reactive bot.

```
GET /api/v1/public/sectors/sector-1/events?limit=20
GET /api/v1/public/sectors/sector-1/events?limit=20&since=2026-05-12T15:14:32.456Z
```

Query parameters:

- `limit` — number of events (default 20, max 100).
- `since` — ISO-8601 timestamp; only events with `accepted_at > since`.

Response (200):

```json
[
  {
    "x": 100,
    "y": 200,
    "color": 3,
    "accepted_at": "2026-05-12T15:14:32.456Z",
    "chunk_version_after": "17",
    "bot_name": "m25-conway"
  }
]
```

Events are sorted descending by `id` (matches `PixelEvent` insertion order). `chunk_version_after` is a stringified BigInt. `bot_name` is included so a reactive bot can identify the author without making a second roundtrip; `bot_id`, `owner_id`, `api_key_id`, and `request_id` are intentionally not exposed.

#### Live viewers

Approximate active viewer count over a rolling ~2-minute window. Used by the in-repo `m25-visitor-pulse` bot and useful for any bot that wants to react to audience size.

```
GET /api/v1/public/sectors/sector-1/viewers
```

Response (200):

```json
{
  "active": 42,
  "window_seconds": 120
}
```

`active` counts unique client IPs that hit any `/api/v1/public/sectors/...` endpoint in the current or previous minute (so a viewer's count drops within ~2 minutes of them closing the tab). The number is directional, not exact: bot egress IPs are counted, NAT collapses many users into one, and the bucketing is wall-clock-minute rather than rolling. All biases are bounded and small at our launch scale.

#### Polling cookbook

This is what the in-repo viewer does. Adapt freely for third-party scrapers.

```js
// Once at startup
const meta = await fetch('/api/v1/public/sectors/sector-1').then(r => r.json());
const cache = new Map(); // key: "cx,cy" → { version, bytes }

// Every 1s
async function tick() {
  const manifest = await fetch('/api/v1/public/sectors/sector-1/manifest').then(r => r.json());
  for (const { chunk_x, chunk_y, version } of manifest) {
    const k = `${chunk_x},${chunk_y}`;
    const cached = cache.get(k);
    if (cached && cached.version === version) continue;
    const headers = cached ? { 'If-None-Match': `"${cached.version}"` } : {};
    const res = await fetch(
      `/api/v1/public/sectors/sector-1/chunks/${chunk_x}/${chunk_y}`,
      { headers },
    );
    if (res.status === 304) continue;
    const bytes = new Uint8Array(await res.arrayBuffer());
    cache.set(k, { version: res.headers.get('X-Chunk-Version'), bytes });
    // ...repaint chunk...
  }
}
setInterval(tick, 1000);
```

End-to-end target: pixel write → visible-on-screen ≤ 2s. Worst-case under the cache config above is ~2s; typical is ~1.2s.

### Operator endpoints

`POST /api/v1/admin/revoke-key` — revoke any bot key by id. Gated by `ADMIN_TOKEN` (sent as `Authorization: Bearer <admin-token>`). Missing or wrong token returns `404 not_found` rather than 401, so the path's existence isn't advertised. Failed admin-auth attempts also write an `AdminAuditEvent` row (`action: "failed_admin_auth"`) so an attempted compromise leaves a durable trail.

The endpoint is **idempotent**: calling revoke on an already-revoked key returns `200 { revoked: true, idempotent: true, revoked_at: <original-time> }` rather than 404 or 409. This means an operator can re-run a revoke after a network blip without worrying about state. Each call writes a fresh `AdminAuditEvent` row regardless (so the audit trail captures every attempt), with `payload_json.idempotent: true` distinguishing no-op calls from state-changing ones.

CLI parity: `pnpm admin:revoke-key <key-id>` (with `ADMIN_TOKEN` and optionally `BOTPLACE_URL` in process env). Every admin endpoint has an equivalent `pnpm admin:*` script — adding an admin endpoint without one is treated as incomplete.

## Examples

### Agent bootstrap (shell-only, end-to-end)

After a one-time browser sign-in at `https://botplace.app/`, mint a Personal Access Token from `/bots`. Then everything else can run from a shell loop — no browser, no DevTools.

```bash
# Drop these in your shell once. The PAT is owner-scoped (full bot
# management); treat it like a password.
export BOTPLACE_PAT='bp_pat_…'
export BOTPLACE_URL='https://botplace.app'

# 1. Mint a bot (returns the bot id + first key plaintext, ONCE).
RESP=$(pnpm -s bot:create my-shell-bot)
echo "$RESP" | jq .
BOT_ID=$(echo "$RESP" | jq -r .id)
KEY=$(echo "$RESP" | jq -r .api_key.plaintext)

# 2. Write a pixel with the bot key.
curl -fsS -X POST "$BOTPLACE_URL/api/v1/pixels" \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{"sector_id":"sector-1","x":42,"y":7,"color":3}' | jq .

# 3. Read it back. Same bot key works on read endpoints.
curl -fsS "$BOTPLACE_URL/api/v1/sectors/sector-1/pixels/42/7" \
  -H "Authorization: Bearer $KEY" | jq .

# 4. Rotate the key when it gets long-lived.
ROTATED=$(pnpm -s bot:rotate-key "$BOT_ID" "$(echo "$RESP" | jq -r .api_key.id)")
NEW_KEY=$(echo "$ROTATED" | jq -r .plaintext)
```

The `pnpm bot:*` and `pnpm pat:*` scripts are thin wrappers over the corresponding HTTP endpoints — see `scripts/bots/` and `scripts/pat/`. Every owner action has a script; if you find yourself reaching for DevTools, the script is missing and that's a bug.

### Mint a bot via the browser

If you'd rather start from the UI (sign in once, then keep using the browser):

```js
// Mint a bot — returns plaintext key once.
const { id: BOT_ID, apiKey } = await fetch('/api/v1/bots', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'my-bot' }),
}).then(r => r.json());
const KEY = apiKey.plaintext;
console.log('save this key:', KEY);
```

Then in your terminal:

```bash
KEY='bp_live_…'  # paste from above
BASE='https://botplace.app'  # or http://localhost:3000

# Write a pixel
curl -s -X POST "$BASE/api/v1/pixels" \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{"sector_id":"sector-1","x":100,"y":200,"color":3}'

# Confirm it persisted
curl -s "$BASE/api/v1/sectors/sector-1/pixels/100/200" \
  -H "Authorization: Bearer $KEY"
```

### Sector dimensions + palette

Always make this call once at bot startup so you don't hardcode dimensions or palette indices that may change later:

```bash
curl -s "$BASE/api/v1/sectors/sector-1" \
  -H "Authorization: Bearer $KEY"
```

### Bulk read via chunk

```bash
# 10000-byte body = 100 × 100 pixels, each byte is a palette index
curl -s "$BASE/api/v1/sectors/sector-1/chunks/0/0" \
  -H "Authorization: Bearer $KEY" \
  -o chunk-0-0.bin -D headers.txt
grep '^x-chunk-' headers.txt   # version + updated_at
```

### Handling `429`

```bash
RESPONSE=$(curl -s -w '\n%{http_code}' -X POST "$BASE/api/v1/pixels" \
  -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" \
  -d '{"sector_id":"sector-1","x":0,"y":0,"color":0}')
STATUS=$(printf '%s' "$RESPONSE" | tail -n1)
if [ "$STATUS" = "429" ]; then
  # Read Retry-After from headers — pause and try again.
  ...
fi
```

The bot bucket fills at 1 token / 60s per key, so the practical strategy is "send at most one write per minute per bot key, regardless of `Retry-After`."
