# M2.5 launch-bots probe

Manual deployment + verification steps for the M2.5 launch bots (`m25-visitor-pulse`, `m25-sparkle`, `m25-conway`). Once PR #14 merges, run through this in order.

The bots are real bots owned by your personal owner record. They use POWER-tier API keys (provisioned by this probe) and write pixels through `POST /api/v1/pixels` — same path any external bot would hit. Their reads use the public endpoints. They're operator-issued, but indistinguishable from a third-party reactive bot at the audit/log level.

## Phase 1 — Production schema migration (automatic)

`vercel-build` runs `prisma migrate deploy` on every deploy. The M2.5 PR adds `Bot.rate_tier` as a new column with `DEFAULT 'FREE'`, which is safe to apply to a populated table (existing rows backfill automatically). No operator action.

**Verify after Vercel finishes deploying:**

```bash
curl -fsS https://botplace.app/api/v1/public/sectors/sector-1/viewers
# Expect: 200 with {"active": N, "window_seconds": 120}
# Even right after deploy, N may be 0 — middleware only counts the
# current 2-minute window.
```

If the endpoint 404s, the M2.5 routes haven't deployed yet — wait another minute.

## Phase 2 — Provision the three launch bots

This step writes 3 rows to `bots`, 3 to `bot_api_keys`, and 3 to `admin_audit_events` in the production Neon DB. Idempotent: re-running is safe (existing bots get skipped).

**One-time:** pull production env into a shell-only buffer (no committed file).

```bash
# Make sure you're authenticated to Vercel CLI
vercel whoami

# Source production env into a single-use shell session so the seed
# script reads the production DATABASE_URL.
vercel env pull --environment=production /tmp/m25-prod-env
set -a; source /tmp/m25-prod-env; set +a

# Run the seed script. JSON output to stdout includes the three
# plaintext API keys — SHOWN ONCE.
pnpm m25:seed-launch-bots --owner-email travis@folkforest.com | tee /tmp/m25-keys.json

# Tear down the env file IMMEDIATELY so the prod DB URL doesn't sit on disk.
shred -u /tmp/m25-prod-env  # or `rm -P` on macOS
```

Expected JSON output:

```json
[
  { "bot_name": "m25-visitor-pulse", "bot_id": "...", "api_key": "bp_live_...", "provisioned": "created" },
  { "bot_name": "m25-sparkle",       "bot_id": "...", "api_key": "bp_live_...", "provisioned": "created" },
  { "bot_name": "m25-conway",        "bot_id": "...", "api_key": "bp_live_...", "provisioned": "created" }
]
```

Re-running shows `"provisioned": "existed"` for each — no new keys minted.

**Pass criterion:** three bots with `rate_tier=POWER` exist under your owner record. Audit rows exist with `action="set_bot_rate_tier"` and `provisioned_by="m2.5-seed-launch-bots"`.

## Phase 3 — Wire the API keys into Vercel project env

The cron routes read each bot's key from process env. Three env vars to set in Vercel project → Settings → Environment Variables (Production scope):

| Variable | Value |
|---|---|
| `M25_VISITOR_PULSE_KEY` | The `api_key` for `m25-visitor-pulse` from Phase 2's JSON |
| `M25_SPARKLE_KEY` | The `api_key` for `m25-sparkle` |
| `M25_CONWAY_KEY` | The `api_key` for `m25-conway` |
| `BOTPLACE_API_BASE_URL` | `https://botplace.app` (optional; defaults to this) |

`CRON_SECRET` is automatically set by Vercel when cron jobs are configured — no manual action.

**Tear down the local file:** delete `/tmp/m25-keys.json` after copying the keys.

**Redeploy** the production deploy so the cron routes pick up the new env vars (the `vercel.json` cron config triggers a deploy on PR merge; the env-only change needs a manual redeploy via Vercel dashboard → Deployments → ⋯ → Redeploy).

## Phase 4 — Manually trigger each cron route once

Vercel cron fires every minute. To verify before waiting, trigger each route manually with curl + the CRON_SECRET. You can read CRON_SECRET from the Vercel dashboard once the cron jobs are deployed.

```bash
CRON_SECRET='...'   # from Vercel project env
URL=https://botplace.app

echo '=== visitor-pulse ==='
curl -fsS "$URL/api/cron/visitor-pulse" \
  -H "Authorization: Bearer $CRON_SECRET" | jq .

echo '=== sparkle ==='
curl -fsS "$URL/api/cron/sparkle" \
  -H "Authorization: Bearer $CRON_SECRET" | jq .

echo '=== conway ==='
curl -fsS "$URL/api/cron/conway" \
  -H "Authorization: Bearer $CRON_SECRET" | jq .
```

**Pass criterion:** each cron route returns a JSON body with the bot's name and a `pixels_written` count > 0 (visitor-pulse may write 0 on a cold canvas with no viewers; sparkle may skip with `"skipped": true, "reason": "no_non_self_event"` if nothing has been written recently; conway should write the seed pentomino on a cold chunk).

If a route returns 404 with `{"error": "not_found"}`, the `Authorization` header is wrong — double-check `CRON_SECRET`.

If a route returns 500 with `error_slug=missing_bot_key`, the corresponding `M25_*_KEY` env var isn't set on the deployment.

## Phase 5 — Wait one minute, refresh the viewer

```bash
open https://botplace.app/
```

Within 1–2 minutes after Phase 3, Vercel's cron should fire all three routes once. You should see:

- **Top row** of the canvas (y=0, y=1): a yellow "viewer meter" — at least a few 2x2 blocks lit, accounting for your view (which the middleware counts).
- **Somewhere on the canvas:** scattered pixel writes from the conway bot (R-pentomino in whichever chunk corresponds to the current minute-of-hour mod 100), evolving over the next 60 minutes as conway rotates through chunks.
- **Eventually:** off-white sparkle halos around recent writes from sparkle bot.

The canvas is now perpetually alive. Open `https://botplace.app/?debug` to see the 2×2 visitor-pulse blocks distinctly with the magenta grid overlay.

## Phase 6 — Verify via structured logs

In the Vercel deployment dashboard (Functions → Logs), filter for `path:"/api/cron/visitor-pulse"` (and the other two) over the last 10 minutes. You should see:

- One invocation per minute per route.
- `bot_name` field matches the route.
- `pixels_written` field non-zero (when there's work to do).
- No `error_slug` field on healthy ticks.

Also filter for `auth_type:"bot_key"` and `bot_id:<one of the bot ids>` — you should see one `/api/v1/pixels` POST per pixel each bot wrote. These are the same log lines any third-party POWER bot would produce.

## Phase 7 — Verify rate-tier behavior

Pick one of the cron routes' bot_ids, fetch its details, confirm the tier:

```bash
# Authenticated as Travis (e.g. with a PAT), list bots:
BOTPLACE_PAT='bp_pat_...'
curl -fsS https://botplace.app/api/v1/bots \
  -H "Authorization: Bearer $BOTPLACE_PAT" | \
  jq '.[] | select(.name | startswith("m25-"))'
```

Expected: `rate_tier: "POWER"` on each m25-* bot.

## Cleanup / rollback

If something goes wrong and you need to disable the bots:

1. **Quick disable** — remove the three `M25_*_KEY` env vars from Vercel project env and redeploy. The cron routes will fire but return 500 `missing_bot_key` and write nothing.
2. **Full revert** — `gh pr revert <merge-sha>` to back out the PR. The schema migration is forward-compatible; the new column stays, the table is fine. The crons stop firing because `vercel.json` is gone.
3. **Demote individual bots** — `pnpm op admin:set-bot-tier <bot-id> FREE` — bot keeps existing but is throttled to 1/min.
