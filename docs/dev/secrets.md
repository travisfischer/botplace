# Secrets

> **Audience:** the app owner/maintainer (Travis + any operator running this app's infrastructure). Bot authors using Botplace do **not** read this document ‚Äî they manage one secret (their bot's API key), through the bot-author flow, not through any of the patterns here. App-owner secrets and bot-author secrets are two different surfaces with two different stories.

Agent-accessible secrets for this repo live in 1Password.

## Convention

- Vault: `Agents`
- Item: descriptive name; the canonical convention is `<Service>` or `<Service> <Purpose>` (e.g., `Cloudflare API Token`, `Neon`, `OpenAI`).
- Field: `credential` (preferred) or `token`
- Reference form: `op://Agents/<exact item title>/credential`. Item titles are case-sensitive and must match what 1Password shows in the UI exactly.

## Items

Vendor automation tokens for the services this app integrates with. Each token is scoped to the minimum permissions needed for its current automation tasks; broaden only with a corresponding code or doc change that justifies it. Items not yet present should be added before the first script that needs them.

- `op://Agents/Cloudflare API Token/credential`
- `op://Agents/Neon/credential`
- `op://Agents/Vercel/credential`
- `op://Agents/Botplace API Key Pepper/credential` ‚Äî production pepper (32 random bytes) used to HMAC bot API keys + owner PATs. Dev pepper is generated per Neon branch by `pnpm db:bootstrap` and lives only in `.env`.
- `op://Agents/Botplace Auth Secret/credential` ‚Äî Auth.js `AUTH_SECRET` for production session signing.
- `op://Agents/Botplace Google OAuth/credential` ‚Äî Google OAuth client secret. The non-secret client ID is stored alongside as a `client id` field for convenience.
- `op://Agents/Botplace Upstash Redis/credential` ‚Äî Upstash REST token. URL is stored alongside as a `rest url` field. Both are process-env-only ‚Äî never written to `.env`.
- `op://Agents/Botplace Admin Token/credential` ‚Äî static operator token for `/api/v1/admin/...` endpoints.

The three M2.5 launch-bot keys (`M25_VISITOR_PULSE_KEY`, `M25_SPARKLE_KEY`, `M25_CONWAY_KEY`) and the cron secret (`CRON_SECRET`) deliberately do **not** live in 1Password. They're minted by `pnpm m25:seed-launch-bots` directly into Vercel project env (Production scope) ‚Äî operator-scoped, application-managed, and rotatable via `pnpm bot:rotate-key` if exposed. See [`probes/m2.5-launch-bots.md`](probes/m2.5-launch-bots.md).

**Naming convention note:** item titles should not contain `(`, `)`, or other characters that 1Password CLI treats as special in `op://` references. Use `Botplace <Service>` (space-separated, no parens) so refs work as plain `op://Agents/<title>/credential`.

Add an entry when introducing a new secret. If a needed item is missing from 1Password, ask the user to create it rather than inventing a path.

## Source of truth by context

| Context | Source of truth | Notes |
|---|---|---|
| Production / Preview deploy runtime | Vercel project env vars + Vercel‚ÜîNeon integration injections | Already wired. No changes. |
| Long-lived automation credentials in scripts | Process env | Adapters: cloud-agent platform secrets, `op run`, `op read` into a single-use shell var, manual export. Scripts only know about process env. |
| Local Next.js runtime + Prisma CLI | One canonical local env file (`.env`) | Generated by `pnpm db:bootstrap`. Contains only disposable values ‚Äî see allow / deny list below. |
| Public docs / committed templates | Variable names and `op://` refs only | Never plaintext values. |

## Two patterns for sourcing secrets into a shell

The app owner has **two** ways to hand long-lived secrets to a one-shot script. They are not interchangeable ‚Äî each fits a specific context.

### Pattern 1 ‚Äî Local dev with 1Password (`pnpm op <script>`)

```bash
pnpm op db:bootstrap
pnpm op admin:revoke-key <key-id>
```

`scripts/env/op-run.sh` wraps `op run --env-file=op.env -- pnpm <script>`. The committed `op.env` template lists `op://` references; `op run` resolves them through the operator's signed-in 1Password session and injects them into the inner command's process env. Values never touch disk.

Use this pattern when:
- The script targets your **local dev** environment, OR your local-dev `.env` (with disposable per-branch values) is the canonical env for the task.
- The needed secrets are stored in 1Password's `Agents` vault and listed in `op.env`.
- You're on a developer workstation with `op` signed in.

Cloud-agent runtimes don't use this ‚Äî their platform injects process env directly.

### Pattern 2 ‚Äî Production one-shot from operator workstation (`vercel env pull` + `source`)

```bash
vercel env pull --environment=production /tmp/prod-env
set -a; source /tmp/prod-env; set +a
pnpm m25:seed-launch-bots --owner-email <you>
shred -u /tmp/prod-env   # or `rm -P` on macOS
```

Use this pattern when:
- The script must run against **production** with the **exact** production env (DB URL, integration-injected vars like `DATABASE_URL` from the Vercel‚ÜîNeon integration, `KV_REST_API_TOKEN` from Vercel‚ÜîUpstash, etc.).
- The needed secrets live in Vercel project env, not in 1Password (most integration-injected secrets do).
- The host is ephemeral or operator-only ‚Äî never a shared long-running cloud-agent runtime where `/tmp` survives between sessions.

Trade-off: this pattern writes secrets to `/tmp` for the duration of the script. The `shred -u` (or `rm -P`) at the end is non-negotiable.

### Which pattern when?

| Task | Pattern |
|---|---|
| Generate / refresh local `.env` from disposable per-branch values | Pattern 1 (`pnpm op db:bootstrap`) |
| Revoke a leaked key in local dev | Pattern 1 (`pnpm op admin:revoke-key ‚Ä¶`) |
| Mint launch bots in **production** | Pattern 2 (`vercel env pull` + `source`) |
| Run any operator script against **production** | Pattern 2 |
| Any script in a cloud-agent runtime | Neither ‚Äî the runtime's platform-injected process env IS the source |

The cloud-agent path (process env injected by the platform) is a third context, but it has no operator action ‚Äî there's nothing to source manually.

## Allow / deny list for the local `.env`

**Allowed** (regenerated by bootstrap; safe-ish if leaked because the underlying branch is disposable):

- `DATABASE_URL` ‚Äî pooled, scoped to the selected dev branch
- `DATABASE_URL_UNPOOLED` ‚Äî unpooled, scoped to the selected dev branch
- `NEON_BRANCH_NAME` ‚Äî for "what am I touching" sanity in scripts
- `BOTPLACE_API_KEY_PEPPER` ‚Äî disposable per-branch dev pepper. Bootstrap preserves an existing value across re-runs (so previously-minted dev keys stay valid) and generates a fresh value on first run. Production pepper is **never** written here ‚Äî it lives in Vercel project env + 1Password only.
- `AUTH_SECRET` ‚Äî disposable per-branch JWT signing secret for Auth.js. Same lifecycle as the pepper: bootstrap preserves on re-runs, generates fresh on first run. Production `AUTH_SECRET` lives in Vercel project env + 1Password only.
- `GOOGLE_CLIENT_ID` ‚Äî non-secret OAuth client ID. Tied to the production OAuth client in Google Cloud project `botplace-app`. Bootstrap persists it to `.env` from process env (sourced via `op.env` ‚Üí 1Password in the local-streaming setup, or platform-injected in the cloud-agent setup). If you're hand-rolling `.env` without bootstrap, set it manually. Preview/production get it from Vercel project env.
- `KV_REST_API_URL` (or `UPSTASH_REDIS_REST_URL`) ‚Äî non-secret REST endpoint for the Upstash Redis instance (e.g. `https://<id>.upstash.io`). Production + preview get it from the Vercel‚ÜîUpstash integration with no custom prefix; the integration injects `KV_*` names matching Vercel's KV convention. Local dev doesn't need this ‚Äî `lib/rate-limit.ts` falls back to an in-process memory bucket when no Upstash env is set. The TOKEN that pairs with this URL is **not** allowed in `.env` ‚Äî see deny list.
- Non-sensitive feature flags or app config

**Not allowed** (must come from process env, supplied by an external adapter):

- `NEON_API_KEY`, `VERCEL_TOKEN`
- `KV_REST_API_TOKEN` / `UPSTASH_REDIS_REST_TOKEN` (the URL that pairs with it is non-secret ‚Äî see allow list)
- `AUTH_SECRET`, `GOOGLE_CLIENT_SECRET`, `ADMIN_TOKEN` (`GOOGLE_CLIENT_ID` is non-secret ‚Äî see allow list)
- Production `BOTPLACE_API_KEY_PEPPER` (only the disposable dev pepper is allowed ‚Äî see above)
- Any provider key (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, future provider keys)
- Any production credential that is not disposable branch-local connection material

## Access rules

- Fetch with `op read op://Agents/<exact item title>/credential`, assign to a single-use env var, and use it only for the immediate task.
- Never echo secret values to stdout, logs, commit messages, or chat transcripts. Pipe into the consuming command, don't print and re-paste.
- Long-lived automation credentials live in process env, never on disk in repo-tracked files. Bootstrap may write to `.env`, but only from the allow list above.
- If `op` is not signed in, ask the user to run `! eval $(op signin)` themselves ‚Äî do not attempt to authenticate on their behalf.

`pnpm env:check` reports presence of required vars by name without printing values, and exits non-zero if any required input is missing.
