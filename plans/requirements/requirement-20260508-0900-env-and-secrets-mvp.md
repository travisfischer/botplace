# Requirement: Env and Secrets MVP

## Problem / Outcome

Botplace is built in public and is expected to be developed by humans and coding agents, both locally and in cloud environments. Environment variables need to be easy enough for agents to use without hidden setup, but long-lived secrets must not be normalized into plaintext files that can leak through commits, transcripts, or screen-shares.

The desired outcome is a small, coherent convention:

1. **Vercel** is the source of truth for deployed runtime secrets.
2. **Process env** is the universal interface that all scripts read from. Cloud-agent platform secrets, `op` injectors, and manual exports are all just adapters into the same contract.
3. **One canonical local env file** (preferably `.env`) contains disposable branch connection strings and non-sensitive metadata only. Long-lived API tokens never land here.
4. **Bootstrap scripts** read long-lived credentials from process env and materialize only the disposable subset to the local env file.

This requirement complements [cloud-agent-neon-dev-branches](requirement-20260508-0851-cloud-agent-neon-dev-branches.md), which defines the bootstrap that consumes this contract.

## Current State (Botplace)

- `docs/dev/secrets.md` establishes an `Agents` 1Password vault, `credential` field convention, and `op://...` references. It also says: never echo, never write to disk outside process env, ask the user to `op signin`.
- `docs/dev/setup.md` currently documents two env files: `.env.local` for Next.js and `.env` for Prisma (because `prisma.config.ts` calls `dotenv/config`). This split is an MVP liability — it creates app/Prisma drift risk. Target state: one canonical `.env`, both toolchains read it.
- `prisma.config.ts` prefers `DIRECT_URL → DATABASE_URL_UNPOOLED → DATABASE_URL`.
- `lib/prisma.ts` reads runtime DB access from `DATABASE_URL`.
- `.gitignore` already excludes `.env` and `.env.*`.

## Source-of-Truth Table

| Context | Source of truth | Notes |
|---|---|---|
| Production / Preview deploy runtime | Vercel project env vars + Vercel↔Neon integration injections | Already wired. No changes. |
| Long-lived automation credentials in scripts | Process env | Adapters: cloud-agent platform secret UI, `op run`, `op read` into a single-use shell var, manual export. Scripts don't care which adapter populated process env. |
| Local Next.js runtime + Prisma CLI | One canonical local env file (`.env`) | Generated by bootstrap. Contains only disposable values. |
| Public docs / committed templates | Variable names and `op://` refs only | Never plaintext values. |

## Allow / Deny List for the Local Env File

**Allowed in `.env`** (regenerated by bootstrap; safe-ish if leaked because the underlying branch is disposable):

- `DATABASE_URL` (pooled, scoped to the selected dev branch)
- `DATABASE_URL_UNPOOLED` (unpooled, scoped to the selected dev branch)
- `NEON_BRANCH_NAME`
- Non-sensitive feature flags or app config

**Not allowed in `.env`** (must come from process env, supplied by an external adapter):

- `NEON_API_KEY`
- `VERCEL_TOKEN`
- `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` / future provider keys
- Any production credential that is not disposable branch-local connection material

A non-public, non-streaming local session may temporarily write other values to ignored env files — but that is not the documented default and no agent workflow should depend on it.

## Requirements

### Functional Requirements

- [ ] `docs/dev/secrets.md` includes a source-of-truth table or equivalent paragraph and the rule that long-lived credentials live in process env, never on disk in repo-tracked files.
- [ ] `docs/dev/setup.md` documents one canonical local env file as the local happy path for both human and cloud-agent sessions.
- [ ] Bootstrap (defined in [cloud-agent-neon-dev-branches](requirement-20260508-0851-cloud-agent-neon-dev-branches.md)) writes only allow-list values to the local env file.
- [ ] A `pnpm env:check` (or equivalent) command reports: which required env vars are present in process env (by name only), which `op://` refs resolve (by name only), the selected Neon branch name, and DB-OK status. Exits non-zero if any required input is missing.
- [ ] `lib/prisma.ts` (Next.js runtime) and `prisma.config.ts` (Prisma CLI) read the same canonical local env file.

### Non-Functional Requirements

- [ ] Scripts are non-interactive once `op` is signed in or process env is populated.
- [ ] Scripts fail loudly and specifically when a required ref is missing or a `op://` ref does not resolve.
- [ ] Scripts never echo secret values, never log full DB connection strings.
- [ ] Public-facing docs are safe to share and do not enumerate per-token capabilities.
- [ ] The convention does not fight Next.js or Vercel defaults.

## Acceptance Criteria

- [ ] A cloud agent with `NEON_API_KEY` and `NEON_PROJECT_ID` in its platform env can run bootstrap with no 1Password dependency.
- [ ] A human with `op` available can populate the same vars via `op run --env-file <local-ref-template> -- pnpm db:bootstrap` and get the same outcome.
- [ ] `pnpm env:check` prints names and statuses, never values.
- [ ] After bootstrap, `pnpm dev` works and `pnpm db:migrate:dev` works against the same canonical local env file.
- [ ] The generated local env file contains the allowed values and does not contain `NEON_API_KEY`, `VERCEL_TOKEN`, or other long-lived tokens.
- [ ] Vercel production/preview behavior is unchanged.

## Risks and Mitigations

- **Convention feels strict for quick local iteration.** *Mitigation:* scripts read from process env, regardless of adapter. Disposable branch URLs can still be materialized to disk. The strict rule is just: long-lived tokens stay out of `.env`.
- **`.env` vs `.env.local` confusion.** *Mitigation:* canonicalize on `.env`. Update toolchain config so both Next.js local runtime and Prisma CLI read it.
- **DB URLs are still credentials.** Disposable branch URLs aren't cost-free if leaked. *Mitigation:* keep the env file gitignored, mask in logs, keep branches disposable enough that rotation is cheap.
- **Multiple ways to populate process env feels like multiple systems.** *Mitigation:* document them explicitly as adapters into the same contract; scripts only know about process env.

## Implementation Notes

Suggested package scripts (singular surface area, not one per concern):

```json
{
  "env:check": "scripts/env/check-env.sh",
  "db:bootstrap": "scripts/db/bootstrap.sh"
}
```

Suggested doc updates:

- `docs/dev/secrets.md` — add the source-of-truth table and the no-long-lived-secrets-in-`.env` rule.
- `docs/dev/setup.md` — replace the two-file (`.env` + `.env.local`) story with the single canonical `.env` happy path.

## Validation Strategy

- Run bootstrap via cloud-agent process env once, via `op run` once. Confirm both produce equivalent canonical env files.
- `pnpm env:check` against a fresh shell with deliberately missing `NEON_API_KEY` exits non-zero with an actionable message naming the missing var.
- Confirm the canonical env file contains only allow-listed values; grep for forbidden token names in the generated file.
- `pnpm dev` returns DB-OK on `/api/health`; `pnpm db:migrate:dev --name smoke-env` succeeds.

## Possible Future Enhancements

- Committed `op://` ref template (`docs/dev/secret-refs.env.example`) for convenient `op run` invocations.
- 1Password Environments instead of hand-maintained ref files.
- Vercel custom environments for named staging/QA targets.
- CI secret scanning.
- Documented credential-rotation runbook.

## Notes

Streaming-development hygiene (mic-on screen-share scenarios) is a per-developer concern, not a Botplace project concern. See [`docs/dev/streaming-safety.md`](../../docs/dev/streaming-safety.md) for an opt-in checklist that builds on this convention.
