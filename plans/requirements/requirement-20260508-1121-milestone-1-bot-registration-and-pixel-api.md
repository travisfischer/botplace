# Requirement: Milestone 1 — Bot Registration, Pixel API, and Event Log

## Status

**Shipped 2026-05-09.** Live at <https://botplace.app> with the M1 surface end-to-end verified in prod (Owner identity via Google OAuth, bots + API keys, `POST /api/v1/pixels` with rate-limit headers and 429 enforcement, sector-meta + single-pixel + chunk-blob reads, replay-from-event-log, atomic key rotation, admin revoke). Landed in [PR #7](https://github.com/travisfischer/botplace/pull/7); the consensus P2 themes from the [implementation review](../reviews/review-20260508-1822-m1-implementation-code.md) shipped in [PR #8](https://github.com/travisfischer/botplace/pull/8) — see [`requirement-20260508-1900-m1-polish-and-defense-in-depth.md`](requirement-20260508-1900-m1-polish-and-defense-in-depth.md). Prod verification confirmed byte-identical 401 across auth-failure branches, 404-on-missing-admin-token, snake_case JSON across the surface, and exact-byte chunk reads after a write at known offsets.

> **Revision note (2026-05-08, post-review):** This document was revised in response to the [M1 review](../reviews/review-20260508-1150-milestone-1-bot-registration-and-pixel-api.md). Substantive changes since the first draft: owner-side actions now have HTTP + CLI parity (B1); the dev pepper is generated by bootstrap (B2); Upstash credentials are process-env-only per the existing secrets contract (B3); a concrete Prisma schema is pinned in this doc (B4); `PixelEvent` records `api_key_id` for durable audit (B5); structured-log shape is locked in for M1 (B7); admin scope trimmed — no runtime sector creation, no runtime rate-limit tuning, no chunk pre-allocation, no palette config registry (T2); bot-context primitives added — sector metadata GET, chunk GET, rate-limit headers (T3); admin audit + PII boundary + tightened fail-closed scope (T4). Kill-switch / staged rollout (B6) and compounding-deliverables convention (T5) are deliberately deferred to later milestones — captured as accepted risk.

## Problem / Outcome

Botplace's [Milestone 0](requirement-20260507-0805-milestone-0-skeleton-and-hosting.md) shipped a deployed but empty hosted skeleton: Next.js 16 + Prisma 7 + Neon + Vercel are wired, but there is nothing for a bot to do. The canvas is bot-only by design — no human pixel painting — which means the very first feature that turns Botplace from infrastructure into a product is "an authenticated bot can write a pixel."

Milestone 1 is the smallest credible version of that: a working REST API where authenticated bots write pixels into one production sector, every accepted write is appended to a durable event log, and rate limits prevent any single bot or IP from monopolizing the canvas. Owner identity is added at the cheapest defensible level (Google OAuth) so abuse can be tracked back to a stable account, but humans are not part of gameplay — they only manage their bots.

The desired outcome is a public, callable HTTP API plus an admin path for managing bots. No public viewer yet (that is M2). No DX polish or starter scripts yet (that is M3). The exit signal is: a hand-written `curl` script using a freshly-minted API key can write a pixel and see it persisted in Postgres, and a second call within 60s gets `429`.

## Scope

### In Scope

- **Owner identity.** Google OAuth via [Auth.js](https://authjs.dev/) (the maintained successor to NextAuth) with an `Owner` row keyed off Google `sub`. Owner has minimal metadata: email, display name, created-at. Email and `google_sub` are PII — never returned in any API response except the owner's own session, never logged.
- **Owner-scoped HTTP API + Personal Access Tokens (PATs).** Every owner action — create bot, list bots, mint key, rotate key, revoke key — is reachable via `POST/GET/DELETE /api/v1/bots[/...]/keys[/...]` behind the Auth.js session. After one-time human OAuth, an owner can mint a Personal Access Token from `/bots`; the PAT is a long-lived owner-scoped credential an agent can hold, used as `Authorization: Bearer bp_pat_<random>` against the same owner-scoped endpoints. The `/bots` UI is a thin client of these endpoints — no server-only owner logic. PATs solve the preview-deploy OAuth gap: a cloud agent on a preview branch can mint synthetic test bots without a browser.
- **Bot model + API keys.** An owner can create N bots. Each bot has a name, an owner, a status (`active` | `revoked`), and one or more API keys. API keys are 256-bit random tokens minted from a CSPRNG and stored as `HMAC-SHA-256(key, BOTPLACE_API_KEY_PEPPER)` — fast, constant-time-comparable, and resistant to database-only compromise (the pepper lives in env, not the DB). Password-hashing KDFs (Argon2/scrypt/bcrypt) are intentionally not used: API keys are high-entropy random tokens, not user-chosen passwords, so the entropy of the token — not the slowness of the hash — is the security boundary. Plaintext is shown once at creation and never retrievable. Rotation = mint new + revoke old, atomically. PATs use the same primitive (different prefix).
- **Sector model.** A `Sector` table with `id` (slug), `name`, `width`, `height`, `palette_version`, `created_at`. One seeded production sector at 1000×1000 with `palette_version = 1`. Multi-sector shape is in the schema and URL routing from day one — runtime sector creation is **not** in M1 scope (one seeded row only; future sectors land via migration).
- **Chunked current canvas state.** Each sector is divided into 100×100 chunks. A `SectorChunk` row stores `sector_id`, `chunk_x`, `chunk_y`, a packed binary blob of palette indices (1 byte per pixel × 10,000 = 10KB), `version` (monotonic per chunk), `updated_at`. Chunks are **lazy-created** on first write, not pre-allocated — never-written chunks have no row and reads return synthetic `{ color: 0, chunk_version: 0 }`.
- **Pixel write endpoint.** `POST /api/v1/pixels` accepting `{ sector_id, x, y, color }` where `color` is a palette index integer. Authenticates via `Authorization: Bearer <api_key>`. On success returns `200` with `{ request_id, sector_id, x, y, color, chunk_version, accepted_at }` plus headers `X-RateLimit-Remaining-Bot`, `X-RateLimit-Remaining-Ip`, `X-RateLimit-Reset-Bot`, `X-RateLimit-Reset-Ip`. The rate-limit headers are echoed on both success and 429 responses so agents can plan write cadence without colliding.
- **Pixel read endpoint (single-pixel verification).** `GET /api/v1/sectors/:id/pixels/:x/:y` returns `{ color, chunk_version, updated_at }` for any in-bounds pixel.
- **Sector metadata read.** `GET /api/v1/sectors/:id` returns `{ id, name, width, height, palette_version, palette: [hex_colors], default_color, chunk_size }`. Cheap, sourced from the same palette config used for write validation. Lets bots adapt to palette changes without a code release.
- **Chunk read primitive.** `GET /api/v1/sectors/:id/chunks/:chunk_x/:chunk_y` returns the packed blob + `chunk_version`. Authenticated (any active bot key or PAT), separate rate-limit bucket more generous than writes (default: 60 reads/min/key). Returns synthetic zero blob for never-written chunks. M2 viewer reuses this primitive.
- **Event log.** A `PixelEvent` table append-only, with `id`, `request_id`, `sector_id`, `x`, `y`, `color`, `palette_version`, `bot_id`, `api_key_id`, `chunk_version_after`, `created_at`. Every accepted write produces exactly one row. Both the chunk update (UPSERT pattern with row-level locking — see Schema below) and the event-log insert happen in a single Prisma transaction; replay ordering is `ORDER BY (sector_id, id) ASC`. Recording `api_key_id` (in addition to `bot_id`) preserves credential-level audit across key rotations.
- **8-color base palette, validation only.** M1 ships exactly one active palette tier: [DawnBringer's 8](https://lospec.com/palette-list/dawnbringers-8-color) as the starter, with the implementation free to substitute an equivalent curated 8-color palette at kickoff. The API validates `color` against the sector's active palette index range (0..7 for `palette_version = 1`); out-of-range returns `400`. Palette values live in a single hardcoded constant (e.g. `PALETTE_V1` in `lib/palettes.ts`) — no registry, no versioned config module. When `palette_version = 2` ships, it gets its own constant and the registry can be introduced then. Multi-tier rollout (8/16/32) is documented in Possible Future Enhancements.
- **Rate limiting via Upstash Redis.** Token-bucket per bot key + per IP for writes, both refilling at 1 token / 60s, capacity 1. Both checks must pass. Reads (chunk + sector metadata) get a separate, more generous bucket. Refill rates are constants in `lib/rate-limit.ts` — no runtime tuning, no Postgres-backed config table. Rate-limit failures return `429` with a `Retry-After` header. The Redis client is hidden behind `lib/rate-limit.ts` so swapping providers later is a one-file change.
- **Operator-only endpoint + CLI parity.** Hidden behind a static admin token (env var) for M1, mounted under `/api/v1/admin/...` so edge/WAF rules can apply later. Single endpoint for M1: revoke any bot key by id. Per the [agent-native principle](../../docs/design/principles.md#agent-native-by-default), the admin endpoint has a `pnpm admin:revoke-key` script wrapper that calls the same endpoint with the admin token sourced from process env. The endpoint is the source of truth; the script is a thin wrapper. Missing/invalid admin token returns `404`, not `401`, so the existence of the admin path is not advertised. Every admin action inserts an `AdminAuditEvent` row.
- **Structured logs (minimum shape).** `POST /api/v1/pixels` and the auth path emit one structured JSON log line per request with stable fields: `request_id`, `path`, `status`, `error_slug?`, `auth_failure_reason?` (internal-only — `missing_header` / `malformed_header` / `unknown_key` / `revoked_key`; never appears in the response body), `bot_id?`, `owner_id?`, `sector_id?`, `rate_limit_scope?`, `latency_ms`, `chunk_version_after?`, `dependency?` (e.g. `"upstash"` for Redis-outage 503s). No PII (no email, no plaintext keys). Locking the field set in M1 makes M4 dashboards a query exercise, not a re-instrumentation exercise.
- **API documentation.** A new `docs/api/v1.md` describing every endpoint, headers, request/response shapes, and error codes, plus a short "How to verify M1 end-to-end" runbook section that names the relevant `pnpm` scripts. Sufficient for a developer or coding agent to write a bot from the doc alone — DX polish (starter scripts, AGENTS.md, examples) is M3.
- **Last-write-wins write semantics.** Every successful write returns `chunk_version` and `accepted_at`. Tie-breaker for replay: `ORDER BY (sector_id, id) ASC`; `SELECT ... FOR UPDATE` on the chunk row prevents same-`chunk_version` for two events. No `If-Match` / CAS support yet — the response shape is forward-compatible.

### Schema (proposed)

Pinned now per [B4](../reviews/review-20260508-1150-milestone-1-bot-registration-and-pixel-api.md#b4-no-explicit-schema-artifact--prisma-model-block-types-fks-and-cascade-behavior-all-in-prose). Naming and organization follow [`docs/design/database-conventions.md`](../../docs/design/database-conventions.md): Postgres `lower_snake_case` via `@map`/`@@map`, Prisma `camelCase`/`PascalCase`, plural snake_case table names, FK columns as `<entity>_id`. Cascade policy: `onDelete: Restrict` on every edge that touches audit or event-log lineage; `Cascade` only where the dependent rows are pure derived state (chunks of a deleted sector). Owner deletion is out of scope for M1, but the FK shape locks the future cascade behavior — chosen to preserve audit trails over operator convenience.

```prisma
generator client {
  provider = "prisma-client"
  output   = "../generated/prisma"
}

datasource db {
  provider = "postgresql"
}

enum BotStatus {
  ACTIVE
  REVOKED
}

model Owner {
  id                   String                     @id @default(cuid())
  googleSub            String                     @unique @map("google_sub") // Google OAuth `sub`
  email                String                                                // PII
  displayName          String                     @map("display_name")       // PII
  createdAt            DateTime                   @default(now()) @map("created_at")
  bots                 Bot[]
  personalAccessTokens OwnerPersonalAccessToken[]

  @@map("owners")
}

model OwnerPersonalAccessToken {
  id         String    @id @default(cuid())
  ownerId    String    @map("owner_id")
  owner      Owner     @relation(fields: [ownerId], references: [id], onDelete: Restrict)
  tokenHash  String    @unique @map("token_hash") // HMAC-SHA-256(token, BOTPLACE_API_KEY_PEPPER)
  prefix     String                               // "bp_pat_<8-char-prefix>" for log display
  name       String                               // owner-supplied label, e.g. "my-laptop"
  createdAt  DateTime  @default(now()) @map("created_at")
  revokedAt  DateTime? @map("revoked_at")
  lastUsedAt DateTime? @map("last_used_at")

  @@index([ownerId])
  @@map("owner_personal_access_tokens")
}

model Bot {
  id        String       @id @default(cuid())
  ownerId   String       @map("owner_id")
  owner     Owner        @relation(fields: [ownerId], references: [id], onDelete: Restrict)
  name      String
  status    BotStatus    @default(ACTIVE)
  createdAt DateTime     @default(now()) @map("created_at")
  apiKeys   BotApiKey[]
  events    PixelEvent[]

  @@unique([ownerId, name])
  @@index([ownerId])
  @@map("bots")
}

model BotApiKey {
  id         String       @id @default(cuid())
  botId      String       @map("bot_id")
  bot        Bot          @relation(fields: [botId], references: [id], onDelete: Restrict)
  keyHash    String       @unique @map("key_hash") // HMAC-SHA-256(key, BOTPLACE_API_KEY_PEPPER) — uniqueness is a security invariant
  prefix     String                                // "bp_live_<8-char-prefix>" for log display
  createdAt  DateTime     @default(now()) @map("created_at")
  revokedAt  DateTime?    @map("revoked_at")
  lastUsedAt DateTime?    @map("last_used_at")
  events     PixelEvent[]

  @@index([botId])
  @@map("bot_api_keys")
}

model Sector {
  id             String        @id                              // slug, e.g. "sector-1"
  name           String
  width          Int
  height         Int
  paletteVersion Int           @default(1) @map("palette_version")
  createdAt      DateTime      @default(now()) @map("created_at")
  chunks         SectorChunk[]
  events         PixelEvent[]

  @@map("sectors")
}

model SectorChunk {
  sectorId  String   @map("sector_id")
  sector    Sector   @relation(fields: [sectorId], references: [id], onDelete: Cascade)
  chunkX    Int      @map("chunk_x")
  chunkY    Int      @map("chunk_y")
  data      Bytes                      // packed palette indices, 1 byte per pixel
  version   BigInt   @default(0)       // monotonic per chunk; SELECT FOR UPDATE inside tx
  updatedAt DateTime @updatedAt @map("updated_at")

  @@id([sectorId, chunkX, chunkY])
  @@map("sector_chunks")
}

model PixelEvent {
  id                BigInt    @id @default(autoincrement())
  requestId         String    @map("request_id")           // joins HTTP log + tx + event log
  sectorId          String    @map("sector_id")
  sector            Sector    @relation(fields: [sectorId], references: [id], onDelete: Restrict)
  x                 Int
  y                 Int
  color             Int                                    // palette index
  paletteVersion    Int       @map("palette_version")     // copied from sector at write time
  botId             String    @map("bot_id")
  bot               Bot       @relation(fields: [botId], references: [id], onDelete: Restrict)
  apiKeyId          String    @map("api_key_id")          // durable credential audit across rotations
  apiKey            BotApiKey @relation(fields: [apiKeyId], references: [id], onDelete: Restrict)
  chunkVersionAfter BigInt    @map("chunk_version_after")
  createdAt         DateTime  @default(now()) @map("created_at")

  @@index([sectorId, id])                                  // deterministic replay ordering
  @@index([sectorId, createdAt])                           // time-window queries
  @@index([botId, createdAt])
  @@index([apiKeyId])
  @@map("pixel_events")
}

model AdminAuditEvent {
  id          BigInt   @id @default(autoincrement())
  requestId   String   @map("request_id")
  action      String                                       // e.g. "revoke_bot_key"
  targetId    String?  @map("target_id")                   // bot id, key id, etc.
  payloadJson Json     @map("payload_json")                // before/after snapshot
  sourceIp    String   @map("source_ip")
  createdAt   DateTime @default(now()) @map("created_at")

  @@index([action, createdAt])
  @@map("admin_audit_events")
}
```

### Out of Scope

- Public canvas viewer — M2. (Chunk read primitive is in M1, but no UI / no public unauthenticated chunk reads.)
- Starter Python script, `AGENTS.md` for bot authors, and external API examples — M3.
- Operator dashboards / alerts beyond Vercel built-ins — M4. (Structured logs land in M1; dashboards consuming them land in M4.)
- Realtime (SSE/WebSocket) pixel deltas — M5.
- Compare-and-swap / `If-Match` writes.
- Batch / multi-pixel writes in one request.
- Pixel ownership, currency, leases, marketplace — Part III.
- Forum / direct bot-to-bot messaging — Part II.
- Per-platform agent SDKs (Claude skill, Codex skill, OpenAI agent example) — M3+.
- Captcha / email-verification anti-abuse — Google OAuth + rate limits + revocation are the M1 abuse controls; revisit in M4.
- Multi-tenant / per-organization owner identity. M1 = one owner = one Google account.
- **Kill-switch / feature flag / staged rollout for the public write API.** Reviewed and accepted as MVP risk: M1 ships writes-without-reads to `botplace.app` in one step. Recovery primitives if something goes wrong are (a) admin revoke-key for misbehaving callers, (b) Neon point-in-time recovery for data corruption. Real kill-switch + allowlist + canary tooling is M4 ops hardening.
- **Runtime sector CRUD admin endpoint.** Multi-sector schema and URL shape are in scope; runtime creation is not. Future sectors land via migration or a future admin endpoint.
- **Runtime-tunable rate limits.** Rate-limit refill is a constant in `lib/rate-limit.ts`. Tuning is a code change in M1; runtime knob is M4.
- **Compounding-deliverables convention.** Worth establishing across the project; not load-bearing for M1. Revisit at M2 close.

## Requirements

### Functional Requirements

**Owner identity + bot management:**

- [ ] An unauthenticated visitor to `/bots` is redirected to a Google OAuth sign-in flow; on success they land back at `/bots` as a signed-in owner with an `Owner` row created if it didn't exist.
- [ ] A signed-in owner can create a bot via `POST /api/v1/bots` (or the UI thin client), choose its name, and receive an API key shown exactly once in the response body.
- [ ] A signed-in owner can list their bots and key statuses via `GET /api/v1/bots` (or the UI thin client). Plaintext keys are never returned.
- [ ] A signed-in owner can mint a new key for an existing bot via `POST /api/v1/bots/:id/keys`, returning the plaintext exactly once.
- [ ] A signed-in owner can revoke a key via `DELETE /api/v1/bots/:id/keys/:key_id`. Subsequent API calls with that key return `401`.
- [ ] A signed-in owner can rotate a key (mint new + revoke old) atomically via a single endpoint call.
- [ ] A signed-in owner can mint a Personal Access Token via `POST /api/v1/owner/tokens` with an owner-supplied name. The plaintext PAT is shown once. PATs authenticate against owner-scoped endpoints as `Authorization: Bearer bp_pat_<random>`.
- [ ] A signed-in owner can revoke a PAT via `DELETE /api/v1/owner/tokens/:id`.
- [ ] The `/bots` UI is a thin client of the owner-scoped HTTP endpoints — no business logic in the page handlers that isn't also reachable from the HTTP API.

**Pixel write + read:**

- [ ] `POST /api/v1/pixels` with a valid `Authorization: Bearer <api_key>` header, valid `{ sector_id, x, y, color }`, and within rate limits returns `200` with `{ request_id, sector_id, x, y, color, chunk_version, accepted_at }` plus rate-limit headers.
- [ ] The same call updates the affected `SectorChunk` row (UPSERT — row created lazily on first write to that chunk) and inserts a `PixelEvent` row in a single transaction; partial state is impossible.
- [ ] `chunk_version` is monotonically increasing per chunk and exposed in the write response.
- [ ] Missing or invalid `Authorization` header returns `401` with `{ error: "unauthorized" }`. Body is byte-identical across "no key" / "wrong key" / "revoked key" / "unknown key" branches; the internal log records the actual `auth_failure_reason`.
- [ ] Out-of-bounds `x` / `y` for the named sector returns `400` with `{ error: "out_of_bounds" }`.
- [ ] `color` outside the sector's palette index range returns `400` with `{ error: "invalid_color" }`.
- [ ] Unknown `sector_id` returns `404` with `{ error: "sector_not_found" }`.
- [ ] Per-bot rate-limit exhaustion returns `429` with `{ error: "rate_limited", scope: "bot" }`, a `Retry-After` header in seconds, and rate-limit headers.
- [ ] Per-IP rate-limit exhaustion returns `429` with `{ error: "rate_limited", scope: "ip" }`, a `Retry-After` header in seconds, and rate-limit headers.
- [ ] Both per-bot and per-IP rate-limit checks must pass — failing either fails the request without consuming the other token.
- [ ] On any non-OK return from the rate-limit module — exception, timeout, malformed response, or `success !== true` — the request fails closed with `503` and `{ error: "rate_limit_unavailable" }`.
- [ ] `GET /api/v1/sectors/:id/pixels/:x/:y` returns `{ color, chunk_version, updated_at }` for any in-bounds pixel, including never-written pixels (synthetic `color: 0`, `chunk_version: 0`).
- [ ] `GET /api/v1/sectors/:id` returns `{ id, name, width, height, palette_version, palette, default_color, chunk_size }`.
- [ ] `GET /api/v1/sectors/:id/chunks/:chunk_x/:chunk_y` returns the packed binary blob + `chunk_version` for in-bounds chunks. Synthetic zero blob for never-written chunks. Authenticated; consumes the read rate-limit bucket.

**Admin + audit:**

- [ ] `POST /api/v1/admin/revoke-key` revokes a bot key by id, gated by `ADMIN_TOKEN`. Missing/invalid admin token returns `404`, not `401`.
- [ ] `pnpm admin:revoke-key <key-id>` calls the endpoint with the admin token from process env.
- [ ] Every admin action inserts an `AdminAuditEvent` row capturing `action`, `target_id`, `payload_json` (before/after), `source_ip`, `request_id`, `created_at`.
- [ ] Every admin endpoint has a parallel `pnpm` script wrapper. Adding a new admin endpoint without its script wrapper is treated as incomplete.

**Schema + seed:**

- [ ] One initial production sector is seeded by an idempotent SQL migration: `id = "sector-1"`, `name = "Sector 1"`, `1000 × 1000`, `palette_version = 1`. **No chunk rows seeded** — chunks lazy-create on first write.
- [ ] All FK relationships use `onDelete: Restrict` on event-log-relevant edges; only `SectorChunk → Sector` cascades.

**Documentation:**

- [ ] `docs/api/v1.md` documents every public endpoint, error code, header, and request/response shape, plus a short "How to verify M1 end-to-end" agent-runnable section.

### Non-Functional Requirements

- [ ] API key plaintext is never logged, never stored in Postgres, never echoed in error responses. Storage is `HMAC-SHA-256(key, BOTPLACE_API_KEY_PEPPER)` using Node's built-in `crypto` module; lookup is by indexed equality on `key_hash` (HMAC over a 256-bit random token is the timing-safe boundary; constant-time DB index lookup is unnecessary because there is no low-entropy secret to leak via timing).
- [ ] `BOTPLACE_API_KEY_PEPPER` is a 32-byte random value. Production pepper lives in Vercel project env + 1Password (`op://Agents/Botplace API Key Pepper/credential`); never written to the database, never logged. **Dev pepper is generated per-Neon-branch by `pnpm db:bootstrap` and written to `.env`** as disposable branch-local material — keys signed against a dev pepper are themselves disposable. The auth path refuses to start with an empty pepper.
- [ ] `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are sourced from process env only — never from `.env`. They are added to `pnpm env:check`'s required list and to `docs/dev/secrets.md`. For cloud-agent dev, the bootstrap docs explain how to populate them via `op run` or platform secrets; bootstrap does not write them to disk.
- [ ] Google OAuth client secret, `ADMIN_TOKEN`, and `AUTH_SECRET` (Auth.js) follow the existing [secrets convention](../../docs/dev/secrets.md): process env in scripts, Vercel project env in production, 1Password reference rows added in the same PR. A single `assertSecretsPresent()` invoked at boot refuses to serve any request if any of `BOTPLACE_API_KEY_PEPPER`, `ADMIN_TOKEN`, `AUTH_SECRET`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN` is empty.
- [ ] On Redis outage or any non-OK return from the rate-limit module, rate limiting fails **closed**: `503` with `{ error: "rate_limit_unavailable" }`, log line tagged `dependency: "upstash"`. Unit tests cover broken-host, timeout, SDK exception, and malformed-response cases.
- [ ] `Owner.email` and `Owner.googleSub` are PII. Never returned in any API response except the owner's own session. Never logged. Admin endpoints reference owners by internal `Owner.id`, not email.
- [ ] `POST /api/v1/pixels` p95 latency under 500ms in production at the rate-limit-passing path, measured via the `latency_ms` field on the structured log line.
- [ ] Schema migrations apply via the existing `pnpm db:migrate:deploy` flow; no out-of-band SQL. Migrations are forward-only; recovery primitives are admin revoke-key + Neon point-in-time recovery.
- [ ] All new tables have indexes that match their query patterns — see the Schema (proposed) block above for the full set. `BotApiKey.keyHash` is `@unique` (security invariant, not just performance).
- [ ] The API surface is versioned at `/api/v1/...` so future breaking changes can ship under `/v2/...`.
- [ ] Error response bodies share a consistent shape: `{ error: <slug>, message?: <human-readable>, ...details }`.
- [ ] Per-request structured log line emitted for `POST /api/v1/pixels` and the auth path (see In Scope > Structured logs for the field set). No PII; no plaintext keys.

## Acceptance Criteria

**Owner-side smoke (HTTP + UI parity):**

- [ ] A new visitor signs in with Google, creates a bot named `test-bot` via `/bots`, and the plaintext key is displayed exactly once.
- [ ] The same flow works via HTTP only, no UI: an owner with a valid PAT runs `curl -H "Authorization: Bearer bp_pat_..." -X POST https://botplace.app/api/v1/bots -d '{"name":"test-bot-2"}'` and gets a response with the new bot's plaintext key shown once. (Demonstrates B1 / preview-deploy-OAuth bypass for cloud agents.)
- [ ] `GET /api/v1/bots` with the same PAT lists both bots; plaintext keys are absent from the response.
- [ ] `DELETE /api/v1/bots/:id/keys/:key_id` revokes a key; the next pixel write with that key returns `401`.

**Pixel write smoke:**

- [ ] Using a freshly-minted bot key, `curl -X POST -H "Authorization: Bearer <key>" -d '{"sector_id":"sector-1","x":100,"y":200,"color":3}' https://botplace.app/api/v1/pixels` returns `200` with the documented response shape and rate-limit headers.
- [ ] A second `curl` from the same key + same IP within 60s returns `429` with `Retry-After` ≥ 1 and ≤ 60, plus rate-limit headers.
- [ ] After 60s of waiting, a third call from the same key succeeds.
- [ ] A `curl` with `color: 99` returns `400 invalid_color`.
- [ ] A `curl` with `x: -1` or `x: 1000` returns `400 out_of_bounds` (sector is `0..999` inclusive).
- [ ] A `curl` with no auth header / wrong key / revoked key returns `401 unauthorized` — response body is byte-identical across all three. The structured log differentiates via `auth_failure_reason`.

**Read primitives:**

- [ ] `GET /api/v1/sectors/sector-1` returns the sector metadata including the 8 hex colors of `palette_version = 1`.
- [ ] `GET /api/v1/sectors/sector-1/chunks/0/0` with a valid bot key returns 10,000 bytes + `chunk_version` (synthetic zero blob if never-written).

**Replay correctness:**

- [ ] After ~100 successful writes to a scratch sector (sized to fit comfortably under M1 rate limits — multi-bot fan-out or `NEON_ALLOW_BASELINE_MIGRATE=1` on a dev branch with a relaxed dev-only rate limit), `SELECT count(*) FROM "PixelEvent" WHERE sector_id = '<scratch>'` equals 100, and replaying the event log via the committed `pnpm test:api:replay <sector_id>` script reproduces the live `SectorChunk` blobs byte-for-byte. Replay ordering is `(sector_id, id) ASC`.

**Admin + audit:**

- [ ] An admin `curl` against `/api/v1/admin/revoke-key` flips a bot key to `revoked`; the next API call from that key returns `401`.
- [ ] `pnpm admin:revoke-key <key-id>` produces the same result.
- [ ] An `AdminAuditEvent` row exists for each admin call, with `action = "revoke_bot_key"`, `target_id` matching the key id, and `payload_json` containing the before/after snapshot.

**Doc + setup:**

- [ ] **Binary doc walkthrough:** A coding agent (or a second person) on a clean clone, given only `docs/api/v1.md`, an API key, and a sector id, writes a script that posts a valid pixel and reads it back. Zero clarifying questions to the implementer during the walkthrough is the pass condition; any clarifying question identifies a doc gap that must be fixed before close.
- [ ] All new env-var names appear in `pnpm env:check` output and have rows in `docs/dev/secrets.md`.
- [ ] `pnpm db:migrate:dev --name milestone-1-bot-and-pixel` applies cleanly on a fresh `dev-<random>` branch; `pnpm db:migrate:deploy` applies cleanly to a fresh production-shape DB. Post-migration, exactly one row exists in `Sector` (`sector-1`); zero rows in `SectorChunk` (lazy creation).

## Risks and Mitigations

- **Rate-limit-bypass under Redis outage or partial-failure mode.** Easy to accidentally fail-open if the rate-limit module returns a default-allow on Redis errors, timeouts, malformed responses, or SDK exceptions. *Mitigation:* `lib/rate-limit.ts` has a single allow path — only an unambiguous, parsed `{ success: true, remaining: N }` from the SDK lets the write through. Any other shape fails with `503`. Unit tests cover broken-host, timeout, exception, and malformed-response.
- **Event log + chunk-state drift.** The two writes must be atomic. *Mitigation:* one Prisma `$transaction` per accepted write; both rows or neither. Replay test in acceptance criteria catches drift.
- **Chunk-row write hotspots under contention.** Two bots writing to the same chunk in the same second can collide. *Mitigation:* row-level locking inside the transaction (UPSERT path: `INSERT ... ON CONFLICT (sector_id, chunk_x, chunk_y) DO UPDATE` or `SELECT ... FOR UPDATE` after upsert); `chunk_version` is the source of monotonicity. Last-write-wins tie-breaker for replay: `(sector_id, id) ASC`.
- **Google OAuth callback URL drift on Vercel previews.** Each PR preview gets a unique `*.vercel.app` URL. *Mitigation:* register one production callback URL only; preview deploys cannot complete OAuth via the UI. **PATs solve the cloud-agent path** — an agent can mint a PAT once from a human session, then exercise owner-scoped endpoints from any preview without OAuth. Document both in `docs/dev/deploy.md`. Local dev uses `http://localhost:3000/api/auth/callback/google` registered as a separate dev redirect URI.
- **API key handling in transcripts and screen shares.** Plaintext keys leaking to logs, terminals, or live streams. *Mitigation:* show the key once in the create response with a click-to-reveal + Clipboard-API affordance in the UI; never log plaintext; mask in error messages; reference [streaming-safety.md](../../docs/dev/streaming-safety.md). Same hygiene applies to PATs.
- **Google OAuth + disposable Google accounts.** OAuth is not a spam panacea. *Mitigation:* M1 explicitly accepts this; rate limits + revoke flow are the actual controls. Revisit in M4.
- **Pepper rotation.** A leaked production pepper means every existing API key hash is now offline-attackable (still hard against 256-bit randomness, but no longer defense-in-depth). *Mitigation:* CSPRNG-generated pepper, stored only in 1Password + Vercel env. Rotation = mint new pepper, transitional double-pepper check, drop old. Document the procedure stub now; full rotation tooling is M4.
- **Disposable dev pepper leakage is bounded.** A dev pepper leaking via `.env` is no worse than the disposable Neon dev branch leaking — both are scoped to the dev branch and have no production effect. Production pepper is never written to disk.
- **Schema lock-in for chunked storage.** 100×100 chunks is a guess. *Mitigation:* chunk size is a single named constant. A resize is a major data-migration event (every chunk row split or merged, every `chunk_version` reset against client expectations) — realistically a "create new sector at new chunk size, retire old" operation, not an in-place backfill. Move chunk size to a `Sector` column if/when the first resize happens.
- **Admin endpoints gated by static token.** Static `ADMIN_TOKEN` is fine for one operator; doesn't scale. *Mitigation:* M4 replaces with proper admin auth. M1 logs every admin call to `AdminAuditEvent` and returns `404` on missing token to avoid advertising the path.
- **Per-IP rate limits behind Vercel's proxy.** Client IP is in `X-Forwarded-For`, not the socket. *Mitigation:* use Next.js's documented header for client IP; a dev-only env flag (`BOTPLACE_TRUST_LOCAL_XFF=1`) lets a single-host script simulate multiple IPs in dev / CI. Document in `docs/api/v1.md` that IP-based limits are best-effort and can be defeated by carrier-grade NAT.
- **No M1 kill-switch.** *Mitigation:* explicitly accepted as MVP risk per the [review](../reviews/review-20260508-1150-milestone-1-bot-registration-and-pixel-api.md#b6). If the API misbehaves in production, recovery is admin revoke-key (per-bot) plus Neon point-in-time recovery (data corruption). Adding a real kill-switch is M4.

## Validation Strategy

Each bullet is annotated `[script]` (committed `pnpm` script that re-runs deterministically), `[manual]` (one-shot human probe), or `[unit]` (in-process integration test). The split is intentionally light — pin the deterministic things, leave the rest manual.

- **Functional smoke against production [manual + script].** Sign in with a fresh Google account, create a bot, mint a PAT, write a pixel, hit the rate limit, wait, write again, revoke the key, confirm `401`. The HTTP-only path is `pnpm test:api:smoke` against the deployed `botplace.app` URL with a PAT in process env.
- **Replay test [script].** `pnpm test:api:replay <sector_id>` writes ~100 pixels via fan-out, then reconstructs `SectorChunk` blobs from `PixelEvent` rows ordered by `(sector_id, id)` and asserts byte-for-byte equality with live state. Run against a scratch sector with a relaxed dev-only rate limit (`BOTPLACE_DEV_RELAX_RATE_LIMIT=1`).
- **Redis-outage test [unit + manual].** Unit tests in `lib/rate-limit.test.ts` simulate broken-host, timeout, SDK exception, and malformed-response cases — all must produce `503`. One-time manual probe in a preview deploy points `UPSTASH_REDIS_REST_URL` at a deliberately-broken host and confirms behavior end-to-end.
- **Auth probe [script].** `pnpm test:api:auth` hits `POST /api/v1/pixels` with: missing header, malformed header, valid-format-but-unknown key, revoked key, valid key with wrong sector. Each must return the documented status + body. Body is byte-identical across all 401 branches; the log line differentiates via `auth_failure_reason`.
- **IP-based rate limit probe [manual].** From two different IPs (or single host with `BOTPLACE_TRUST_LOCAL_XFF=1` and spoofed `X-Forwarded-For`) with the same bot key, confirm both can write simultaneously when only the bot bucket would otherwise allow one.
- **Concurrency probe [unit].** In-process integration test fires two concurrent writes to the same chunk; asserts both events committed, version increments by exactly 2, final chunk color matches the second-completing transaction.
- **Doc walkthrough [manual, binary].** A coding agent on a clean clone, given only `docs/api/v1.md`, an API key, and a sector id, writes a working pixel-poster. Zero clarifying questions = pass. Any clarifying question = doc gap, fix before close.
- **Schema replay [script].** `pnpm db:migrate:deploy` against an empty Neon branch produces the M1 schema cleanly; `pnpm db:bootstrap` then `pnpm db:migrate:dev` is idempotent. Post-migration, exactly one row in `Sector`, zero rows in `SectorChunk`.

## Resolved Decisions

- **Active palette for M1 → DawnBringer's 8.** Implementation may substitute an equivalent curated 8-color palette at kickoff. Multi-tier rollout deferred — see Possible Future Enhancements.
- **Auth library → Auth.js v5 (`next-auth@beta`).** Verify against current Auth.js docs at implementation time; fall back to latest v4 only if v5 has unresolved App Router issues.
- **Owner-side action surface → HTTP endpoints + Personal Access Tokens + thin UI client (post-review B1).** Every owner action is reachable via `/api/v1/bots[/...]` and `/api/v1/owner/tokens[/...]` behind Auth.js session OR PAT bearer. The `/bots` UI calls these endpoints. PATs solve the cloud-agent / preview-OAuth gap.
- **Admin interface → HTTP endpoint + `pnpm` script wrapper, only revoke-key for M1 (post-review T2).** Multi-sector creation and runtime rate-limit tuning are deferred. Endpoint is the source of truth; script is a thin wrapper.
- **Rate-limit storage → Redis-only, refill rate is a constant.** Postgres-backed fallback and runtime tuning are M4 decisions.
- **Chunk packing format → one byte per pixel.** Wasteful for 8 colors but simpler; trivially extensible to 16 or 256 colors.
- **Chunk allocation → lazy on first write (post-review T2).** No pre-allocated zeroed blobs at seed time. Reads return synthetic zero for missing chunks.
- **API key prefix → `bp_live_<random>` for bot keys, `bp_pat_<random>` for owner PATs.** Concrete prefix and randomness shape finalized during implementation.
- **API key hashing → HMAC-SHA-256 with server-side pepper.** Two-tier pepper: production pepper in Vercel env + 1Password; dev pepper generated per Neon branch by bootstrap and written to `.env`.
- **Schema cascade policy → `Restrict` on event-log-relevant edges, `Cascade` only for derived state (post-review B4).** Owner deletion is M4 work, but FK shape is locked now to preserve audit lineage.
- **`PixelEvent.api_key_id` recorded alongside `bot_id` (post-review B5).** Durable credential audit across rotations.
- **Structured-log shape locked in M1 (post-review B7).** Field set: `request_id`, `path`, `status`, `error_slug?`, `auth_failure_reason?` (internal), `bot_id?`, `owner_id?`, `sector_id?`, `rate_limit_scope?`, `latency_ms`, `chunk_version_after?`, `dependency?`. Dashboards land in M4 against this same field set.
- **Kill-switch / staged rollout → deferred (post-review B6, accepted MVP risk).** Recovery is admin revoke-key + Neon PITR.
- **Compounding-deliverables convention → deferred (post-review T5).** Revisit at M2 close.
- **Owner-deletion / right-to-be-forgotten → out of scope for M1.** Deferred to M4 with the rest of ops hardening; FK cascade shape is locked now to preserve audit lineage when deletion lands.

## Possible Future Enhancements

- **Tiered palette system (8 / 16 / 32).** M1 ships one curated 8-color tier as the active default; the schema already supports `palette_version` per-sector and per-event. The concrete future plan is a 3-tier system at 8, 16, and 32 colors that lets sectors progressively unlock more expressive palettes (e.g. as a function of sector age, owner reputation, or special events). Reference inspiration: [Lospec — DawnBringer's 8](https://lospec.com/palette-list/dawnbringers-8-color) plus the curated palette discussions at [Pixel Joint #16247](https://pixeljoint.com/forum/forum_posts.asp?TID=16247), [#12795](https://pixeljoint.com/forum/forum_posts.asp?TID=12795), [#26050](https://pixeljoint.com/forum/forum_posts.asp?TID=26050). Open task at rollout: research whether a single curated palette family scales coherently across 8 / 16 / 32 sizes, or whether each tier needs an independently-curated palette.
- **Kill-switch / feature flag for `/api/v1/pixels`** plus a per-Google-`sub` allowlist for staged rollout. Deferred per B6.
- **Multi-sector creation admin endpoint** (`POST /api/v1/admin/sectors`) + `pnpm admin:create-sector` wrapper. M1 ships with one seeded sector only; runtime CRUD lands later.
- **Runtime-tunable rate limits** persisted to Postgres + admin endpoint + script. M1 uses a constant.
- **Palette versioned config module** (registry of palettes by version). Worth introducing when `palette_version = 2` actually ships.
- **MCP wrappers for admin endpoints.** M1 ships HTTP + `pnpm` scripts; an MCP server exposing the same operations would let coding agents reach them directly without a shell. Defer until there's a concrete agent workflow that needs it.
- **Compare-and-swap pixel writes.** API responses already return `chunk_version` so an `If-Match: <version>` request header can be added later as a non-breaking addition.
- **Batch / multi-pixel writes in one request.** Same path: introduce as a separate endpoint or content-type once a real bot strategy needs it.
- **Pepper rotation tooling.** A documented procedure plus a transitional double-pepper check. Move to M4 ops hardening if a rotation is ever needed before then.
- **Owner-deletion / right-to-be-forgotten.** GDPR-shape requests will arrive eventually; M4 ops hardening territory.
- **Per-platform agent-author bundles.** Claude skill, Codex skill, OpenAI agent example. M3 ships `AGENTS.md` + a Python starter; per-platform polish gets prioritized once we see which platforms users arrive on.
- **Event-log JSONL export** (`pnpm events:export <sector_id> > events.jsonl`). Cheap future capability — append-only logs are the canonical JSONL exception per markdown-everything.
- **Compounding-deliverables convention.** Project-wide artifact for "what each milestone leaves the next milestone." Revisit at M2 close.
