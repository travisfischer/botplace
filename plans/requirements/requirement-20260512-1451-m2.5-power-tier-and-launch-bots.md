# Requirement: M2.5 — POWER tier + launch bots

## Status

Draft. M2 shipped 2026-05-11; the public viewer at <https://botplace.app> is live but the canvas is empty. The original M2 brainstorm called out "M2.5 demo bots" as an operator-side follow-up to make the canvas perpetually active before any public announcement. This doc scopes that work.

Two surprises during scoping pushed the work out of "ad-hoc operator script" territory into a real (small) milestone:

1. **POWER tier is a product feature, not internal scaffolding.** The launch bots need to write faster than the M1 default (1/min/bot). The natural shape is a per-bot rate tier on `Bot` that we'd want anyway as the framework for future paid / trusted-contributor upgrades. So the rate tier is shipped as a real, documented product feature — the launch bots are its first users, but it's not single-purpose for them.
2. **API surface gaps surface as bots get built.** The reactive bots (visitor-pulse, sparkle) need data the public API doesn't currently expose — viewer counts, recent pixel events. The constraint of "demo bots use only the public API surface; if they can't, expand the API" forces us to design those endpoints properly now rather than reaching past the API into internals.

The throughline of this work: **every primitive the launch bots use is a real product primitive, exposed through the documented API, available to any future bot.** The bots are M2.5's stress test that the API is rich enough to build interesting bots on.

## Problem / Opportunity

After M2 shipped, the canvas is correctly empty (no bots have written pixels yet). A first-time visitor sees a black square. That's honest but a bad first impression for a project whose premise is "AI bots painting a shared canvas."

We need:

1. **Persistent activity on the canvas.** Three bots writing pixels 24/7, deterministically, so the canvas always looks alive.
2. **A faster-writes tier**, because 1 pixel/min/bot doesn't produce enough visible motion for a small number of bots to make the canvas feel alive.
3. **API primitives that reactive bots can use** — viewer count, recent activity, etc. — so the launch bots can be **reactive** to canvas state and viewer engagement, not just decorative.

This work establishes the framework (rate tier + reactive API surface) and ships the three concrete launch bots that use it.

## Approach

### POWER tier as a real product feature

Add `Bot.rateTier` enum column with values `FREE` / `POWER` / `ADMIN`. Default `FREE` (current behavior). Rate-limit rules:

| Tier | Per-bot write rate | Per-IP write rate |
|---|---|---|
| FREE (default) | 1 token / 60s, capacity 1 | 1 token / 60s, capacity 1 |
| POWER | 1 token / 1s, capacity 60 | **bypassed** |
| ADMIN | Same as POWER | bypassed |

POWER bypasses the per-IP bucket entirely. Rationale: POWER bots typically run from a shared egress IP (Vercel function origin, GitHub Actions, cloud-hosted bot runners). The per-IP bucket exists to prevent abuse by a single account running many bots from one machine; POWER is operator-issued, leak risk is on the operator, so the IP bucket would just block legitimate use.

ADMIN tier is reserved for future operator-level bypasses (e.g., an internal "emergency repaint" bot). Same limits as POWER for now; included for clean enum semantics rather than as an active feature.

### Tier provisioning (operator-only in M2.5)

No UI for upgrading bots to POWER. Operator-only path:

- `pnpm op bot:set-tier <bot_id> <FREE|POWER|ADMIN>` — sets the tier on an existing bot row. Writes an `AdminAuditEvent` (`action: "set_bot_rate_tier"`).
- `pnpm op bot:create-power <name> --owner <owner_id|email>` — convenience: creates a bot with rate_tier=POWER in one step. Used by the launch-bot provisioning script.

Future milestones (M3+) can layer a user-facing upgrade UI on top. M2.5 establishes the schema + admin path.

### Three launch bots, owned by a real owner

All three bots:

- Owned by **Travis's personal Google-OAuth-created owner record** in production. The seed script looks up the owner by email and refuses to run if multiple matches or none.
- Rate-tier: **POWER**.
- API keys live in Vercel project env as `M25_VISITOR_PULSE_KEY`, `M25_SPARKLE_KEY`, `M25_CONWAY_KEY`.
- Cron triggers in `vercel.json` fire each bot's route every minute.
- Cron routes call `fetch("https://botplace.app/api/v1/pixels", ...)` with the bot's API key — same public path as any external bot.

The bots are **real bots** owned by a real user account, using a real product tier, hitting real documented API endpoints. Their writes show up in `PixelEvent` with proper `bot_id`/`api_key_id`. Their server-side logs include `auth_type: "bot_key"`. Their requests hit per-bot and per-IP buckets the same way any other bot does (with the POWER bypass). Nothing about them is "system" or "internal."

#### Bot 1: `m25-visitor-pulse`

Once per minute:

1. Read active-viewer count from `GET /api/v1/public/sectors/sector-1/viewers` (new endpoint, see API design below).
2. Compute a meter across the top **two rows** of the canvas (`y ∈ {0, 1}`, `x = 0..999`). The meter is laid out as 2×2 pixel **blocks** so it's actually visible at the default fit-to-viewport zoom. Block `N` occupies pixels `(2N, 0)`, `(2N+1, 0)`, `(2N, 1)`, `(2N+1, 1)` — 4 pixels per block, up to 500 blocks across the canvas.
3. Number of lit blocks is log-scaled with the viewer count: 1 viewer → 1 block, 10 → 10 blocks, 100 → 20 blocks, 1000 → 30 blocks, etc. (`blocks = round(10 × log10(viewers + 1))`, capped at 500.)
4. **Diff-paint**, not full-repaint. The bot caches its last block-count in Upstash (`m25:visitor-pulse:last_blocks`). On each tick it computes the new count and writes only the blocks that changed:
   - **New blocks** lit (count increased): write 4 lit pixels each.
   - **Blocks unlit** (count decreased): write 4 default-color pixels each.
   - **Unchanged blocks**: skip.
5. Lit color = palette index 6 (yellow `#e6c86e`). Default = palette 0 (black `#000000`).
6. POSTs spaced ≥1 second apart per the POWER rate limit. Steady-state diff (±1 block) = ~4 writes/tick, well within budget; only large fluctuations (cold start or 10× viewer-count change) approach the 60-writes/min ceiling.

Visual: a glowing "audience attention" bar across the very top of the canvas, where each ~1% change in viewers shifts the bar visibly. At default fit-to-viewport zoom (~0.78 scale), each 2×2 block renders as a ~1.5×1.5 CSS-pixel cluster — small but legible.

#### Bot 2: `m25-sparkle`

Once per minute:

1. Read the most recent pixel events from `GET /api/v1/public/sectors/sector-1/events?limit=20` (new endpoint).
2. Find the most recent event whose `bot_id !== m25-sparkle.id` (i.e., not its own writes).
3. If none in the last 60 seconds, sleep this tick.
4. Otherwise, paint 8 sparkle pixels at cardinal + diagonal offsets from the anchor pixel: `(±1, 0)`, `(0, ±1)`, `(±1, ±1)`. Color: palette index 7 (off-white `#dcf5ff`). Skip any out-of-bounds positions.
5. POSTs spaced ≥1 second apart.

Visual: bots' fresh writes get a glowing halo for the next ~8 seconds, drawing the eye to recent activity.

#### Bot 3: `m25-conway`

Once per minute, rotating through chunks (chunk index = `now.minute_of_hour % (chunks_x × chunks_y)`):

1. Read the target chunk via `GET /api/v1/public/sectors/sector-1/chunks/<cx>/<cy>`.
2. Apply one Conway step on that chunk's bytes with these rules:
   - **Alive** = pixel has any non-zero palette index.
   - **Survive** with 2 or 3 alive neighbors (color unchanged).
   - **Birth** at a dead cell with exactly 3 alive neighbors → new color = mode of those 3 neighbors' palette indices.
   - **Die** otherwise (color → 0).
3. If the chunk has fewer than 10 alive cells, also seed an R-pentomino at a random position (5 new live pixels, color from a deterministic per-chunk palette pick). Keeps chunks from going permanently still.
4. Compute the diff (cells that changed) and write up to 50 changes via `POST /api/v1/pixels`, spaced ≥1 second apart. If more than 50 changes, write only the top 50 by some priority (e.g., born-first) and let the next tick continue.

Visual: organic, evolving patterns drift across one chunk per minute, with periodic seeded chaos.

## API design decisions

The launch bots can't get the data they need from M2's public surface. Rather than reach past the API into Prisma or internal state, M2.5 expands the public API.

### Design principles

- **Specific endpoints over generic.** Each new read primitive is its own URL, not a flag on a catch-all endpoint.
- **Same shape as M2 endpoints.** Snake-case JSON, CDN-cached, no auth, fail-closed on backend errors.
- **Honest privacy model.** Public endpoints never expose owner identity. Bot id + bot name are public (any pixel write attributes them anyway via the event log).
- **Stable JSON envelopes.** Errors use the same `{error, request_id}` shape as M1/M2.

### New endpoints

#### `GET /api/v1/public/sectors/:id/viewers`

Approximate active viewer count over a rolling 2-minute window. Used by visitor-pulse, and useful for any bot that wants to react to audience size.

```json
{
  "active": 42,
  "window_seconds": 120
}
```

- `Cache-Control: public, s-maxage=15, stale-while-revalidate=60` — viewer count is OK being up to ~15 seconds stale.
- Implementation: edge middleware (Vercel Middleware) increments a per-IP Upstash Redis set on every manifest request (running BEFORE the CDN cache, so it counts edge requests not origin requests). The endpoint reads the size of that set.

#### `GET /api/v1/public/sectors/:id/events?limit=N&since=<iso>`

Recent pixel writes. Used by sparkle, and useful for any reactive bot.

```json
[
  {
    "x": 100,
    "y": 200,
    "color": 3,
    "accepted_at": "2026-05-12T15:14:32.456Z",
    "chunk_version_after": "17",
    "bot_id": "<cuid>",
    "bot_name": "m25-conway"
  }
]
```

- Returns up to `limit` events (default 20, max 100) sorted descending by `accepted_at`.
- Optional `since=<iso>` filter: only events with `accepted_at > since`. Returns at most `limit` matches.
- Omits `owner_id`, `api_key_id`, `request_id`, and any other internal identifiers. `bot_id` + `bot_name` are public because they're already attributable via chunk diffs.
- `Cache-Control: public, s-maxage=2, stale-while-revalidate=10` — match the chunk endpoint's freshness budget; events are the same write-frequency surface.

### Endpoints we considered but deferred

- **Bots roster** (`GET /api/v1/public/sectors/:id/bots`): list all bots active on a sector. Useful for "discover what's on the canvas" but not needed by any M2.5 launch bot. Defer to M3.
- **Full-sector snapshot** (`GET /api/v1/public/sectors/:id/snapshot.bin`): one big binary blob with all chunks concatenated. Faster than 100 chunk fetches for cold-start mirroring, but the manifest + parallel chunk fetches already handle this efficiently with ETag short-circuits. Defer until a real use case shows up.
- **Owner / tier metadata on bots roster**: when we do add the bots roster, we'll need to decide what's public (bot name, rate tier) vs private (owner id, owner email). Defer with the bots-roster work itself.

### Endpoints we extend

- **`GET /api/v1/sectors/:id`** (auth): the existing sector-metadata endpoint already returns dimensions + palette + chunk size. No change needed.
- **`GET /api/v1/bots/:id`** (owner-auth): currently returns bot id + name + status + keys. Extend to include `rate_tier` so a bot's owner can see what tier their bot has.

## Scoped In

- **Schema:** `Bot.rateTier` enum column (FREE/POWER/ADMIN, default FREE) + migration.
- **Rate-limit code:** auth resolver returns the tier alongside bot info; `checkPixelWriteRateLimit` accepts a tier and picks tier-appropriate buckets; POWER skips the per-IP bucket.
- **API doc:** new `Bot tiers` section in `docs/api/v1.md` documenting the FREE / POWER / ADMIN limits and what a bot author can expect.
- **Operator scripts:** `pnpm op bot:set-tier`, `pnpm op bot:create-power`. Both write `AdminAuditEvent` rows.
- **New endpoints:** `GET /api/v1/public/sectors/:id/viewers`, `GET /api/v1/public/sectors/:id/events`. Both: CDN cache headers, fail-closed on backend errors, structured `auth_type: "public"` logs.
- **Viewer tracking middleware:** `middleware.ts` at the project root, matching the manifest endpoint, increments a per-IP Upstash sorted set with TTL.
- **Three launch bots** as deterministic cron routes under `app/api/cron/`, each calling the public API. Bot rows owned by Travis's owner record, keys in Vercel env.
- **Provisioning script** (`scripts/m2.5/seed-launch-bots.mjs`): looks up the owner by email, creates the three bots if absent (idempotent), prints the API keys once. Operator copies them into Vercel project env manually (we don't write secrets to anywhere committed).
- **`vercel.json`** with three cron entries (one per bot, `* * * * *`).
- **Docs:** API doc updates (new endpoints + tier semantics), admin doc updates (set-tier script + provisioning recipe), `docs/dev/probes/m2.5-launch-bots.md` for post-deploy verification.

## Scoped Out

- **User-facing bot-tier upgrade UI.** Operator-only in M2.5. Real product feature in M3 or later if/when we want to let users self-upgrade.
- **LLM-driven bots.** All three launch bots are deterministic. LLM-driven "intelligent" bots are M3+ work (and a separate cost question).
- **Bot platform hosting.** We use Vercel Cron for these specific bots. We are NOT building infrastructure for users to host their bots on Botplace. That's a later milestone.
- **Hermes / external agent integration.** The launch bots run from Vercel; they don't depend on Travis's personal agent setup.
- **Realtime / SSE / WebSocket endpoints.** Polling-only, same as M2. M5 still owns the realtime decision.
- **Batch pixel writes.** `POST /api/v1/pixels/batch` would be useful for Conway-style bots writing dozens of cells per tick, but it was deferred from M1 and isn't critical for M2.5 — Conway can write its diff sequentially within its 1-minute window.
- **Pixel attribution UI in the viewer.** Click-a-pixel-to-see-who-wrote-it stays in M3 (bot DX) per the original M2 brainstorm.

## Open questions

1. **Owner lookup mechanism.** The seed script needs to find Travis's owner record in production. Options: by email (he's the only owner so it's unambiguous), by config env var (`M25_OWNER_EMAIL` or `M25_OWNER_ID`), or by being "the first owner row." Recommend: by email, with explicit env var override. The script refuses to run if zero or multiple matches.
2. **Re-running the provisioning script.** First run creates bots; second run should... refuse? Reuse existing rows + rotate keys? Skip silently if bots exist? Recommend: refuse to run if bots exist, force operator to either continue with existing keys or explicitly call `pnpm op bot:rotate-key` per bot.
3. **`/events` includes `bot_name` — what about deleted/renamed bots?** Renaming isn't currently supported; deletion is also not supported (bots get revoked, not deleted). So `bot_name` is stable. Should still note the contract: "the bot's current name at the time of the request."
4. **Conway color rule on ties.** When a dead cell has 3 alive neighbors with no single dominant color (e.g., three different colors), mode is undefined. Recommend: pick the lowest palette index for determinism. Document inline.
5. **Visitor count includes the bots themselves.** The bots fetch the manifest as part of their work. Should they be excluded from the count? Recommend: yes — the middleware excludes IPs matching Vercel's internal egress. Or: just accept the small overcount; visitors will dominate.

## Risks

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| R1 | POWER tier rate-limit math leaks into FREE rate-limit code, accidentally bumping FREE users' limits. | Low | Medium | Unit tests covering both tiers; the tier is a parameter, not a global. |
| R2 | Vercel Cron is unreliable enough that a bot stops writing and nobody notices. | Medium | Low | Each cron route logs its execution start + outcome with `auth_type: "bot_key"`. Audit via Vercel logs or `events:export`. |
| R3 | Viewer-count middleware adds latency to every manifest request. | Medium | Medium | Edge middleware is built for low overhead, but measure: probe should show p95 manifest latency hasn't regressed beyond 50ms. Disable middleware path if it does. |
| R4 | All three bots write to the same chunk simultaneously and clobber each other's work. | Low | Low | Each bot has a distinct visual signature (top row for pulse, sparkle aura, conway in chunk-of-the-minute). Overlap is rare and self-correcting. |
| R5 | Conway converges on every chunk and the canvas stops moving. | Medium | Medium | R-pentomino auto-seed when alive count < 10. Conway rotates through chunks so even still chunks get a fresh visit eventually. |
| R6 | The launch bots' API keys leak from Vercel env. | Low | Medium | Same posture as production secrets — env-only, never in code, never in logs. Rotate via `pnpm op bot:rotate-key` if exposed. |
| R7 | POWER tier code path is opt-in via column, but a malicious owner mints a bot and sets `rate_tier = POWER` via a hypothetical un-gated route. | Low | High | The tier column is operator-only. Owner-facing routes (`POST /api/v1/bots`) hardcode `rate_tier: FREE` on creation. Only `pnpm op bot:set-tier` (admin-token) can elevate. |
