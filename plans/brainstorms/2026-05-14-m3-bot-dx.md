M2.5 shipped 2026-05-12. The post-merge review cleanup arc completed yesterday: A-tier in [PR #16](https://github.com/travisfischer/botplace/pull/16), B-tier in [PR #18](https://github.com/travisfischer/botplace/pull/18), and the viewer CDN-cache fix in [PR #19](https://github.com/travisfischer/botplace/pull/19). The canvas is live and visibly active — three deterministic launch bots (`m25-visitor-pulse`, `m25-sparkle`, `m25-conway`) paint patterns once a minute via Vercel Cron. They use the public API (`/viewers`, `/events`, `/chunks/...`), proving the public surface is rich enough to build interesting bots on. That's the M2.5 throughline that pulls into M3: if the operator can build three bots from the public API, an external author should be able to too — **once they have docs, a starter, and an authoring contract.**

M3 is the milestone where Botplace stops being "the operator's app the operator can ship bots into" and becomes "an app any developer or coding agent can write a bot for."

## Problem / Opportunity

Today, writing a Botplace bot requires cloning the repo and reading `docs/api/v1.md` (439 lines), understanding the auth model, the rate-limit shapes, the palette, the chunk addressing scheme, and the optimistic-concurrency contract — then writing your own HTTP client. The launch bots prove this is achievable, but they were written by the same person who wrote the API. There is currently no:

- **Hosted public bot-author docs.** Nothing at botplace.app that an author (or an author's AI agent) can read to bootstrap from zero. The API doc lives in the repo, not the app. A human can't share the URL with someone; an LLM agent can't be pointed at it without auth and a clone.
- **Agent authoring contract.** Nothing analogous to `AGENTS.md` that tells an AI agent — coding agent (Claude Code / Cursor / Codex) or chat-agent (ChatGPT / Claude.ai) — the rules of writing a Botplace bot: how to handle keys, respect rate limits, pick a palette index, not retry into a 429 storm, etc.
- **Palette as a first-class concept in the docs.** The palette is buried inside the API ref ("color is a palette index 0..7 for `palette_version = 1`"). Both authors *and* (future) attribution UIs need a referenceable surface — "this pixel is palette v1 color 3, here's what that means visually."
- **Cross-owner identity.** Bot `name` is unique per owner ([`@@unique([ownerId, name])`](prisma/schema.prisma)). Once non-operator bot authors arrive, two owners will both name a bot `painter` and the public-attribution surface will have no way to disambiguate. The M2.5 B3 decision was: move to a globally-unique `handle` + a non-unique `displayName`. M3 is the natural moment.
- **Pixel attribution surfaces.** The `PixelEvent` table records `bot_id` + `api_key_id` + timestamps for every write. The viewer doesn't show it. The public API can't query it per-pixel. Click-a-pixel-to-see-the-bot is the most-asked-for feature when someone watches the canvas; the data is already there.

M3 closes those gaps. The exit signal: **an LLM agent, given only the URL to Botplace's hosted bot-author docs, can build and ship a working third-party bot to Botplace in under an hour, with no out-of-band help from the operator.**

The non-goals are equally important: M3 is **authoring**, not **hosting**. Botplace will not run other people's bots in M3. There's no in-product editor, no managed runtime, no LLM-as-a-service wired to a bot row. The vision doc's "the platform also offers a default place to host one" lands much later. M3 is the equivalent of "here's a great REST API and the world-class hosted docs to call it" — the bot itself runs on the author's laptop, their own Vercel project, their own server.

## What we've learned since the MVP brainstorm wrote the M3 scope

Eight days have passed since M3 was first scoped. Five things changed.

### 1. The public API surface is much richer than M3 was scoped against

The MVP brainstorm's M3 scope was written when only the authenticated read endpoints existed. Since then M2 added the public manifest / chunk / metadata surface, and M2.5 added `/events` and `/viewers`. The launch bots use **only** the public read surface. This means:

- A starter bot doesn't need authenticated reads to do anything interesting. The starter can be (and should be) "write your bot against the public API; you only need authenticated writes."
- The AGENTS.md should treat reads-via-public and writes-via-bot-key as **the** canonical pattern, not as an alternative to authenticated reads.

### 2. We have working precedent for three bot shapes

The three launch bots — [`visitor-pulse-logic.ts`](src/launch-bots/visitor-pulse-logic.ts), [`sparkle-logic.ts`](src/launch-bots/sparkle-logic.ts), [`conway-logic.ts`](src/launch-bots/conway-logic.ts) — are three different bot archetypes:

- **Reactive bot** (visitor-pulse): reads `/viewers`, writes a derived pattern. Tight feedback loop with the viewer.
- **Ambient bot** (sparkle): writes random decorative pixels around a moving center. Stateless, decorative.
- **State-machine bot** (conway): reads recent state, computes next state, writes diffs. Closest to "non-trivial gameplay logic."

These map naturally to "starter examples." The launch-bot logic files are well-factored — extracted from the runner, testable in isolation. M3 starter examples can be deliberate ports of these to user-runnable starters (without coupling to Vercel Cron).

### 3. The agent-native principle has a sharper edge than the brainstorm captured

[`docs/design/principles.md`](../../docs/design/principles.md) crystallizes "agent-native by default" with both *bot-author* and *operator-contributor* sides. The bot-author side has been mostly aspirational so far — the operator side is what M0–M2.5 hardened. M3 is when the bot-author side becomes the *primary user surface*. That means:

- **AGENTS.md for bot authors lives in this repo** (the same repo bots talk to). It's the canonical authoring contract. The MVP brainstorm's "in-repo Markdown" stance was right; we just need to be specific about *where* in the repo it lives. (See open question Q3.)
- **The starter script must be runnable by a coding agent end-to-end** without a UI step. That rules out anything that requires clicking "Generate API Key" in a web flow — there must be a `pnpm bot:create` + `pnpm bot:mint-key` path for the agent itself, which we already have.

### 4. Handle/displayName isn't optional anymore

M2.5 review B3 documented the decision but deferred implementation. The trigger condition was "M3 first introduces non-operator bot authors." That's now. Specifically:

- **Public attribution surfaces (click-to-inspect, bots roster, `/events`) need a globally unique identifier.** Today `bot_name` collides across owners. The first non-operator to register a bot called `painter` breaks the attribution UX.
- **`/events` already exposes `bot_name`** (per M2.5 resolution #3). Migrating that to `handle` is now a breaking change to the public API. Better to do it in M3 before any third party depends on the current shape.

### 5. The "operator-adjustable rate limits" item from MVP scope is now a smaller problem than scoped

The MVP brainstorm wrote this when there was no tier system. M2.5 shipped FREE/POWER and the operator-elevation path (`PUT /api/v1/admin/bots/:id/tier`). The remaining "operator-adjustable" gap is:

- Per-bot override beyond FREE/POWER (e.g., a bot that needs 5× POWER's rate for a specific demo). No real demand yet.
- Per-sector tuning of FREE/POWER bucket values (e.g., a quieter sector might tolerate higher FREE). No second sector exists yet.

**Decision (2026-05-14, Travis):** drop "operator-adjustable rate limits" from M3 scope entirely. MVP defaults are good enough. Defer to M4 (ops hardening) where it joins the other operator-knobs work, or to whenever the first real demand surfaces.

### 6. The deliverable shape is hosted public docs, not a runnable starter

**Travis's reframe (2026-05-14):** "I want the docs to be beautiful pages that are public on the app itself, not just in the repository." And on the Python starter: "Is that pushing this more towards people thinking of bots as automation vs old-fashioned automation versus AI agents?"

The shape we converged on:

- **Beautiful hosted pages** at botplace.app (not just repo markdown). First-class app surface — same visual care as the viewer.
- **Copy-to-markdown affordance** so an author can ship the doc content into their agent's context window in one click. Both human-readable rendered pages and agent-fetchable markdown sources from the same URLs.
- **"Just point your agent at the URL"** is a supported flow. Authors say to their agent: "Read https://botplace.app/agents.md and build me a bot that does X." That should work without further setup.
- **The "starter" is the agent-authoring contract**, not a runnable script. The author's AI agent generates the code. The docs are the contract; the agent is the codegen.
- **Three valid runtime shapes** for bots, called out explicitly in the docs:
  - **Pure deterministic** — cron-shaped, no LLM at runtime. Cheapest. Launch bots fit here.
  - **Hybrid** — deterministic execution code, with periodic LLM "strategy review" updates regenerating the deterministic code. Recommended default for non-trivial bots.
  - **Full LLM-per-tick** — every action decided by an LLM call. Most expressive, most expensive. Document but don't recommend as the default.

### 7. The palette is load-bearing in M3 in a way the MVP brainstorm didn't anticipate

The palette surfaces in three places now, not one:

- **Bot author docs** — authors need to know what color index 3 means without inferring from hex. Embedded visualization in the docs.
- **Future attribution UIs** (Theme B click-to-inspect) — when a user clicks a pixel, the modal will show "palette v1, color 3 — orange" and link to a palette explanation page. The palette page needs to exist standalone, deep-linkable.
- **Future multi-palette rollout** — `palette_version = 2` is documented as a future tier expansion. The palette page URL scheme must support multiple coexisting palettes from day one.

Travis (2026-05-14): "this may actually deserve its own page separate from the bot instructions, but it probably should be included in the bot instructions." Both: standalone page at `/palettes/<version>` (per Q11), embedded into the bot-author docs.

### 8. Key-handling foot-guns deserve a dedicated docs section

**Travis (2026-05-14):** "We should be really thoughtful about the instructions to recommend best practices that don't set people up for leaking or losing their keys, since that's a common foot gun with these AI agents."

The growing pattern: developers paste API keys into LLM chats so the LLM can "use them." Agent providers vary in how they handle prompt content (logging, training data redaction, conversation persistence). There is no universal "your key is safe in chat" guarantee. The docs must lead with this rather than bury it.

Specific foot-guns to call out:

- **Never paste an API key into an LLM chat session.** Keys belong in process environment variables on the host running the bot, never in conversation context. If you need your agent to test against the live API during authoring, mint a *separate* short-lived key for that, then revoke it.
- **Never commit a key to a repo.** The agent authoring contract should include "if you write code, ensure `.env` is gitignored and reference keys only as `process.env.BP_API_KEY`."
- **One key per host.** Laptop key ≠ Vercel project key ≠ CI key. Easier to revoke a single leaked one.
- **Rotate liberally.** Document `POST /api/v1/bots/:id/keys` (mint new) + `DELETE /api/v1/bots/:id/keys/:keyId` (revoke old). Also document the atomic-rotate path if useful.
- **401-check on startup.** A bot should hit a cheap authenticated endpoint once at startup and fail loud if the key is invalid, instead of burning through retries.

This becomes a `## Key handling` H2 inside the agent authoring contract, *and* a callout block at the top of the Quick Start.

### 9. Glossary + decisions from 2026-05-14 brainstorm review round 2

Travis's second pass through this brainstorm hit terminology that hadn't been spelled out. Decisions captured inline so they don't get lost.

#### Click-to-inspect (Theme B, confirmed in scope)

Minimal interaction model: when the viewer is zoomed in enough that individual pixels are clearly distinguishable, a **single left-click on a pixel** opens a small info box showing `handle`, `displayName`, `written_at`, and the palette color (linking to `/palettes/<version>#color-<i>`). No hover state, no rich card, no follow-this-bot affordance — just attribution. Richer "hub hopper" interactions (chained bot-to-bot browsing, history scrubbing, etc.) are explicitly future-not-now.

#### Pixel attribution endpoint shape (resolved 2026-05-14)

The original M2 Out-of-Scope list called out *two* deferred attribution endpoints — a single-pixel read and a separate single-pixel-event endpoint. **Resolved:** there's only one. The single-pixel read response denormalizes the latest event's attribution into the read body:

```json
{
  "x": 487, "y": 123,
  "color": 3, "palette_version": 1,
  "bot_handle": "m25-conway",
  "bot_display_name": "M25 Conway",
  "written_at": "2026-05-14T15:23:01.234Z"
}
```

That's everything click-to-inspect needs. If pixel history ("everyone who's ever painted this pixel") becomes a real ask later, it slots in as `GET /api/v1/public/sectors/:id/pixels/:x/:y/events` (plural, list-shaped) with the same cache headers as the sector-level `/events` endpoint.

#### `screenToWorld()` — what it does

Viewer-internal helper, not a deliverable in its own right. The viewer renders the 1000×1000 canvas with pan + zoom. Browser click events arrive with **screen** coordinates (where the user's cursor was in the viewport, e.g., `clientX=683, clientY=412`). `screenToWorld(683, 412)` reverses the current pan/zoom transform and returns the **world** coordinate — the actual pixel on the canvas, e.g., `(x=487, y=123)`. Without it, click-to-inspect can't know which pixel to look up. ~10 lines of trig. Speculatively built in M2, deleted in M2.5 review for being speculative; M3 restores it with click-to-inspect as the real caller.

#### Firewall-rules-as-code (`pnpm admin:firewall`) — what it does, and why it's dropping

Today Vercel WAF rules (rate-limit ceilings, IP blocklists, etc.) are managed through the Vercel dashboard — pure click-ops, no code path. The proposed `pnpm admin:firewall` would read a committed TypeScript config declaring the desired rules and reconcile against Vercel's REST API (`--dry-run` to preview the diff, `--apply` to write). Value: bootstrap firewall config on a fresh Vercel project, audit rules in PR review, give agents a non-dashboard path. **Not bot-author-facing.** **Decision (2026-05-14):** drop from M3 + M3.5. Belongs in M4 (ops hardening) alongside other operator-knob work.

#### Wire-protocol nits — what each one means

The M2 review carried three small HTTP-correctness items into M3 polish. Each is independent.

- **X-Request-Id on success responses.** Today success responses include `request_id` in the JSON body (per M2.5 work) but the response *header* (`X-Request-Id: <uuid>`) isn't always set. Asymmetric with error responses. Matters because: tools like `curl -v` and browser devtools surface headers prominently — a bot author seeing a strange-but-200 response wants the request id without parsing JSON. ~5 lines to fix. **Recommend (2026-05-14): in scope.** Lands with Theme A docs work since we're touching the API anyway.

- **RFC-7232-compliant If-None-Match parser.** Bots polling for chunk updates send `If-None-Match: "<version>"` to get a cheap `304 Not Modified` when nothing changed. RFC 7232 says the header can take several spec-correct forms (single quoted, comma-separated list, weak tags `W/"..."`, `*`). Our M2 parser may not handle them all — a bot using a strict HTTP library could send a valid header we don't recognize, falling back to a full 200 (wasted bandwidth, burns rate limit). **Recommend (2026-05-14): defer to M3.5.** Risk is real but speculative until an external bot trips on it. ~30 lines to fix when needed.

- **Vary header on cached responses.** When a CDN caches a response that varies by request header (e.g., `Accept-Encoding` for gzip), the `Vary` response header tells the cache "key by these headers too." Without it, the CDN can serve a gzipped body to a client that asked for identity → corrupt response. Need `Vary: Accept-Encoding` at minimum on cached public endpoints. **Recommend (2026-05-14): defer to M3.5.** Defensive correctness, no known live bug.

#### `AdminAuditEvent.actor_kind` — what it would do

The `AdminAuditEvent` table records admin actions (revokes, tier elevations, etc.). Today the `actor` field is a single string (an email or admin token id). `actor_kind` would add a normalized category alongside: `"admin_token"`, `"oauth_owner"`, `"seed_script"`, `"cron"`, etc. Why it matters: as audit queries grow ("show me everything seed scripts did last week"; "filter cron noise from this audit dump"), normalized categorization beats parsing the actor string. **Today there is one actor kind** (admin_token via the operator). **B8 adds a second** (seed_script with `--actor`). So we're going from one → two actor kinds in M3 — a natural moment to introduce the column.

**Recommendation (2026-05-14):** small forward-compat add, rides with B8 since we're already touching audit-emitting code. ~half a day. But also: fine to wait for a third actor kind. See Q13.

## Themes — how M3 organizes

Three coherent themes. **All three ship together in M3** (Q1 resolved: single milestone). No M3.5 split.

### Theme A: Bot-Author Onboarding

The ground-floor goal: an author with an LLM agent can hand the agent a URL and ship a working bot in under an hour. The deliverable is **a polished, publicly hosted set of bot-author pages at botplace.app** that *both* humans read for understanding *and* AI agents fetch for codegen.

The bar:

- **Beautiful hosted pages**, not just rendered repo markdown. First-class app surface — same care budget as the viewer. The docs are part of the product.
- **Agent-fetchable markdown**. Per-page `?format=md` + a master `/agents.md` aggregator file ([llms.txt convention](https://llmstxt.org)). Plus a "Copy as markdown for your agent" button on each page. See Q9 for the exact UX mix.
- **Self-contained Quick Start.** One page, top-to-bottom, that takes someone from zero (sign in with Google → mint API key → write first pixel) with copy-pasteable commands.
- **Three runtime shapes** explicitly framed (pure deterministic / hybrid / full LLM-per-tick) with cost trade-offs, so authors don't default to "expensive LLM-per-tick" or "boring cron script" by accident. Hybrid is the recommended default.

Concrete pages:

- **`/build`** (root, see Q3) — overview + Quick Start + navigation.
- **`/build/agents.md`** (or top-level `/agents.md`) — the agent authoring contract. **The centerpiece.** A single markdown file an author drops into Claude Code / Cursor / ChatGPT / their custom agent loop and says "build me a Botplace bot that does X." Includes: API surface summary, authentication shape, rate-limit semantics, key-handling foot-guns, palette reference, three runtime patterns, common gotchas (don't retry into 429s; back off on `Retry-After`; read palette from `/api/v1/sectors/:id` rather than hardcoding hex).
- **`/build/quickstart`** — five paragraphs max, copy-pasteable, gets the human from zero to first pixel.
- **`/build/api`** — full API reference, ported from `docs/api/v1.md` with consistent vocabulary, per-section "copy as markdown" buttons.
- **`/build/key-handling`** — the foot-guns, prominent. Also linked-to as a callout from Quick Start and the agent authoring contract.
- **`/palettes/1`** — the active palette page (see Q11 for URL). Standalone, embedded into bot docs, deep-linkable from future attribution UIs ("you're looking at palette v1 color 3 — orange"). Visualizes each color with index, hex, and contextual name. Future palettes (v2, v3) get sibling URLs.

What we're explicitly **not** shipping in Theme A:

- **A runnable starter script** (Python, TypeScript, Node, or otherwise). The framing is: the AI agent writes the bot. The docs are the contract; the agent is the codegen. We are not shipping pre-baked scripts that imply "automation thinking."
- **Per-platform polish** (Claude skill, Cursor rule packs, MCP server, OpenAI Agent template). MVP brainstorm decision stands — defer until we see which platforms users arrive on. (See Q8 if MCP shape pulls itself back in.)
- **Operator-adjustable rate limits.** Moved out of M3; MVP defaults are fine. Tier elevation via the existing admin path is the escape valve.
- **A hosted bot runtime.** Authors run bots wherever they want. Docs mention two patterns (laptop, your own Vercel cron) as worked examples; we don't run anything for them.

#### Illustrative code snippets in the docs (not "the starter")

The docs include short (10–20 line) snippets demonstrating canonical patterns. **Three languages (Q10 resolved):** curl, TypeScript, Python — each pattern gets three side-by-side snippets. Snippets are reference patterns for authors' agents to adapt, not shippable runnable code. Framed as inspiration, not prescription.

Three canonical bot-archetype patterns, lifted from the launch bots:

- **Reactive** (read `/viewers` or `/events`, derive next write). Three snippets (curl, TS, Python).
- **Ambient** (deterministic pixel-painter with no input, like sparkle). Three snippets.
