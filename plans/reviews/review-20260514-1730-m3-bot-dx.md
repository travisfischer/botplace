## Deviations from the requirement

1. **`?format=md` URL syntax** replaced with a dedicated `/api/build-md/<slug>` route. Equivalent intent (raw markdown source per page); cleaner Next App Router shape. Documented in the follow-ups; could be added later via a route handler that intercepts the page request, if there's a real consumer asking for the URL syntax.

2. **MDX → react-markdown.** The requirement called out colocated MDX under `app/build/`. After installing the MDX deps and starting the wire-up, two things became apparent: (a) the only "MDX components" the requirement specifically called out (CopyMarkdownButton, PaletteSwatch, SnippetTabs) collapsed to either layout-level concerns (button) or single-page concerns (palette page in plain TSX) or the R3 plain-code-block fallback (snippet tabs); (b) keeping markdown as TS string constants gave us the same single-source-of-truth property that MDX promised, with simpler wiring and a smaller bundle. Net: dependency footprint dropped (removed `@next/mdx` + `@mdx-js/loader` + `@mdx-js/react`; added `react-markdown` + `remark-gfm`). Same UX, smaller surface.

3. **Click-to-inspect "see recent activity"** opens the raw `/api/v1/public/bots/:handle/events` JSON in a new tab. The requirement says the affordance is "backed by" that endpoint but doesn't specify presentation. Acceptable for M3 minimum scope; flagged in follow-ups as a candidate for a polish PR (render the recent events as a small list inside the inspect box, or build a `/bots/:handle` page).

4. **Migration split** is two SQL files, not one big one. Per R1 mitigation: an operator can apply migration 1 in production, run probe 14, then apply migration 2 (the irreversible drop). Both apply cleanly to the dev branch in one go via `prisma migrate deploy`.

## Code quality notes

- **Tests:** 260 pass, 1 skipped (DB-conditional). 14 new integration tests for the M3 endpoints (`tests/api/m3-attribution-endpoints.test.ts`); 33 new unit tests for handle validation; 5 new pure-math tests for `screenToWorld`.
- **Typecheck:** clean.
- **Lint:** 1 pre-existing warning (`_k` in `tests/rate-limit/fail-open.test.ts`); no new warnings introduced by this PR.
- **Auth privacy posture preserved:** the new attribution endpoints return only `bot_handle` + `bot_display_name`, never `bot_id` / `owner_id` / `api_key_id`. This was the M2.5 stance and the M3 requirement R5 ("Attribution endpoints leak more than intended") expected vigilance; it's been kept.
- **`X-Request-Id` omission on auth failures:** intentional, per the existing `unauthorized()` helper's pattern (avoids amplifying probe attempts). Documented inline in route-helpers.
- **No new permissions surface:** every new endpoint is public-read or owner-auth using the existing resolver. The admin tier endpoint already existed; M3 only adds `actor_kind` to its audit row.
