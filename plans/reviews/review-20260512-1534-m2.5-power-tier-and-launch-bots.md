## Executive Summary

The architecture is principle-aligned. POWER tier is shipped as a real product feature with cleanly factored layers (Prisma enum → tier-aware rate limit → admin-only elevation path → agent-runnable CLI). The launch bots dogfood the public API exactly as the throughline demands. M1/M2 patterns compound forward without reinvention (admin-auth shape, audit-row conventions, CDN-cached public reads, CLI-parity-for-every-admin-endpoint). The schema migration is purely additive and safe to deploy against production unchanged.

The findings cluster around three themes:

1. **Verification gaps.** The cron routes contain non-trivial pure logic (Conway step, log-scale meter math, partial-progress accounting, palette tie-break) but have zero automated tests. The 146-line tier rate-limit test asserts burst capacity but never the **refill rate** that defines POWER. The probe doc is operational but is not a merge gate.

2. **Rollout has soft spots.** Cron auto-fires the moment Vercel rolls the deploy, but bot API keys won't be wired into env until the operator finishes Phase 2–3. Result: a documented "expected interim 500" window with no automated signal. Cross-layer traceability also breaks at the cron → `POST /api/v1/pixels` hop because `request_id` isn't propagated.

3. **Documentation drift surfaced during the rescope.** The requirement still references CLI command names that were renamed before merge (`bot:set-tier` → `admin:set-bot-tier`, `bot:create-power` → `m25:seed-launch-bots`). The frontmatter still reads `status: draft`. The brainstorm artifact M0/M1/M2 each have is missing. M2's forward references to "M2.5 demo bots" were never updated to reflect the rescope into a real milestone.

## P2 — Address before declaring M2.5 shipped

### P2.1 — Cron route logic ships with zero automated tests

**Reviewers:** quality-first F1, release-and-rollout F2

The three cron routes contain pure deterministic logic that is exactly the shape that benefits from unit tests:

- `conwayStep` (Conway step + palette-aware tie-break to lowest index)
- `maybeSeed` / `seedColorForChunk` (R-pentomino auto-seed)
- `viewersToBlocks` (the log-scale math in visitor-pulse)
- visitor-pulse partial-progress accounting (`newLastBlocks = previousBlocks + Math.floor(written / 4)` when only `toLight` was being walked)
- Sparkle's non-self anchor selection + bounds clipping

Probe doc pass criterion ("pixels_written > 0") doesn't catch boundary bugs: 3-way tie in Conway, partial-write resumption in visitor-pulse, "20 of the last 20 events are self" in sparkle.

**Fix:** Extract the pure helpers to a testable module (e.g. `src/launch-bots/<bot>-logic.ts`) and add unit tests for each boundary. None require database or network. ~1.5 hours total.

### P2.2 — POWER tier refill rate is not tested

**Reviewer:** quality-first F2

The defining property of POWER vs FREE is the **refill rate** (1 token / 1s vs 1 / 60s), not burst capacity. The current test (`tests/rate-limit/tier.test.ts:49-68`) asserts the 60-burst-then-fails contract but never advances simulated time to assert the 61st write succeeds 1s later. A regression that swapped `refillIntervalMs: 1_000` → `60_000` (capacity 60, refill 60s) would pass all 6 tier tests but silently break every M2.5 bot.

**Fix:** Add `vi.useFakeTimers()` + time advancement to one POWER test that exhausts the bucket, advances ~1100ms, asserts next write succeeds. Add a symmetric FREE test asserting *no* refill at 1100ms but refill at 61_000ms.

### P2.3 — Cron auto-fires before bot keys are wired; expected 500 noise window

**Reviewer:** release-and-rollout F1

`vercel.json` ships three cron entries (`* * * * *`). Vercel auto-deploys on merge, and crons fire the moment the new build is live — before the operator has finished Phase 2 (seed bots) and Phase 3 (wire `M25_*_KEY` env vars). During that window every minute produces three `error_slug: "missing_bot_key"` log lines at 500 status. The probe doc names this as expected interim state but there's no enforcement mechanism, no banner, no automated signal.

**Fix:** Pick one of three options:

1. **Demote 500 → 200** with `{"skipped": true, "reason": "not_yet_provisioned"}`. Cron is intentionally idle until provisioning lands; failing closed is correct, but 500 is the wrong shape for the documented interim state.
2. **Gate with `M25_BOTS_ENABLED`** env flag. Operator sets it after Phase 3 to wake the bots. Soft-launch posture without splitting the PR.
3. **Split `vercel.json` into a follow-up PR** that ships *after* Phase 2+3 completes. Cleanest "Goldilocks rollout scope" framing.

Recommend option 2 — it's the smallest code change and keeps the rollout sequence explicit.

### P2.4 — Cross-layer traceability breaks at the cron → pixel-write hop

**Reviewer:** observability F1

Each cron tick mints a `requestId`, but `writePixel()` makes a fresh HTTP POST to `/api/v1/pixels` that mints its **own** unrelated request_id. The response's request_id is discarded by `WritePixelResult`. When a cron logs `first_error: "rate_limited"`, an operator triaging "why is conway stuck" cannot pivot from the cron tick to the corresponding `/api/v1/pixels` log line.

**Fix:** Forward a correlation header (`X-Botplace-Parent-Request-Id: <cron-requestId>`) from `writePixel()` to `/api/v1/pixels`; have the pixels route log it as `parent_request_id` when present. Surface the server's response `request_id` into the cron's `WritePixelResult.error` path. ~30 minutes.

### P2.5 — Edge middleware swallows Upstash failures with no log

**Reviewers:** observability F2, quality-first F3

`middleware.ts`'s catch block swallows pipeline errors silently. A sustained Upstash outage drifts viewer counts to 0 within 2 minutes, which drives visitor-pulse to darken its entire meter — a user-visible regression with no log line. The middleware uses `console.warn` for the missing-env path but doesn't `log()` failed writes, and doesn't tag `dependency: "upstash"` (a field `LogFields` defines specifically for this triage axis).

**Fix:** Inside the catch block, `console.warn` with a stable tag and `err.message` (`log()` from `lib/log` doesn't work on the edge runtime — it requires Node APIs, so `console.warn` is the right tool here). Rate-limit the warn to one-per-isolate to avoid spam.

### P2.6 — Edge middleware has no Upstash timeout

**Reviewer:** security-and-privacy F1

`lib/rate-limit.ts` wraps every Upstash call in `withTimeout(p, 2_000)` for exactly this reason: "a hung connection that never resolves would block the request thread until Vercel's invocation timeout fires." The new middleware drops that invariant. Because it runs on every public-sector request (manifest poll, chunks, /events, /viewers), a degraded Upstash region would degrade the entire public read surface, not just the rate-limit downstream.

**Fix:** Wrap the `.exec()` in the same `Promise.race` timeout pattern (or import a shared edge-runtime-safe helper). 500ms is more than enough for a two-op pipeline.

### P2.7 — `BOTPLACE_API_BASE_URL` env override is a credential-exfil/SSRF surface

**Reviewer:** security-and-privacy F5

The launch-bot runner reads `BOTPLACE_API_BASE_URL` from env with no schema check and sends a POWER-tier bearer token in `Authorization` to whatever URL it returns. An operator typo or env-var compromise (where project-env write is broader than DB write) → all three POWER keys exfiltrated on the next cron tick.

**Fix:** Don't accept the override in production. Hard-code `https://botplace.app` and gate any override behind `process.env.NODE_ENV !== "production"`. If preview/staging really needs an override, restrict to Vercel preview/dev scopes and add a startup assertion that the URL ends with `.botplace.app` or is localhost.

### P2.8 — `/events` silently drops events on overflow

**Reviewer:** core/agent-native F1

The events endpoint returns descending-by-id `take: limit` with optional `since` filter. When more than `limit` events occur between polls, the response contains the newest `limit` and silently drops older events. A reactive agent advancing `since` to the newest `accepted_at` it just saw will skip past the dropped events entirely. No `has_more`, no `truncated`, no `next_cursor`. Lossless reactive agents are not composable from the current primitive.

**Fix:** Switch to `since_id`-cursor pagination with `asc` ordering when `since` is provided. Add `has_more: boolean` + `next_cursor` (either response envelope or response headers). Document the recovery pattern. Keep timestamp-based `since=<iso>` as a convenience alias.

### P2.9 — ADMIN rate-tier enum value is speculative build-out

**Reviewer:** goldilocks-scoping F1

ADMIN exists in the Prisma enum, the `BotRateTier` re-aliases, the admin route's `VALID_TIERS`, the API doc, and the rate-limit branching — but has identical runtime behavior to POWER and zero current callers. The requirement itself acknowledges it as "reserved for future." Permanent cost: every future tier-handling reviewer must reason about a third state.

**Fix:** Drop ADMIN from the enum + all three union types + admin route's `VALID_TIERS` + API doc. Re-introduce later (one migration + one tier added) when an "emergency repaint" bot is actually built. ~5 minutes of work.

### P2.10 — Phase 3 env-wiring is dashboard-only with no CLI alternative

**Reviewers:** agent-native F1, cloud-coding F3

The probe doc instructs operators to set `M25_*_KEY` env vars via "Vercel project → Settings → Environment Variables (Production scope)" and redeploy via "Vercel dashboard → Deployments → ⋯ → Redeploy". Both have CLI equivalents (`vercel env add`, `vercel --prod`) but neither is shown. An agent following the probe cannot complete Phase 3.

**Fix:** Rewrite Phase 3 to use `vercel env add M25_VISITOR_PULSE_KEY production < <(jq -r ... /tmp/m25-keys.json)` and `vercel redeploy --prod`. Document `vercel env ls production` for reading back `CRON_SECRET` if needed.

### P2.11 — New env vars not surfaced in `setup.md` / `env:check` / `op.env`

**Reviewer:** cloud-coding F1

A coding agent provisioning a fresh sandbox reads `docs/dev/setup.md`, `scripts/env/check-env.sh`, and `op.env` to learn the contract. None mention `CRON_SECRET`, `M25_*_KEY`, `BOTPLACE_API_BASE_URL`, `m25:seed-launch-bots`, or `admin:set-bot-tier`. The probe doc is excellent but lives outside the discovery surface.

**Fix:** Add a row to `setup.md` "Useful scripts" for `m25:seed-launch-bots` and `admin:set-bot-tier`. Extend `check-env.sh` with a production-only section that flags `CRON_SECRET` + `M25_*_KEY` when in production env. Add a short note to `op.env` or `secrets.md` clarifying that M2.5 bot keys are Vercel-resident, not pulled through `op`.

### P2.12 — Seed script error message points operators at wrong bootstrap for prod

**Reviewer:** cloud-coding F2

The script enforces `BOTPLACE_API_KEY_PEPPER >= 64 chars` with a message saying "Run `pnpm op db:bootstrap` first." That's the **dev-branch** path; following it for prod seeding silently mints keys hashed with the disposable dev pepper, corrupting prod state. The header comment's `Usage:` line (`pnpm op m25:seed-launch-bots`) also contradicts the probe doc's bare `pnpm m25:seed-launch-bots` after sourcing prod env.

**Fix:** Make the error message neutral: "BOTPLACE_API_KEY_PEPPER missing or too short. For local dev: `pnpm op db:bootstrap`. For prod seeding: `vercel env pull --environment=production /tmp/m25-prod-env && set -a; source /tmp/m25-prod-env; set +a`." Fix the header `Usage:` line to match the probe.

### P2.13 — Provisioning playbook is workstation-only

Phase 2 requires `vercel whoami` + `vercel env pull`. `setup.md` doesn't list Vercel CLI as a prerequisite. `op.env` has no `VERCEL_TOKEN` entry (though `secrets.md` line 14 says the item exists). A cloud agent asked to "ship M2.5" cannot complete the probe end-to-end from the docs.

## P3 — Carry into M3 polish window

P3 findings are quality nits, bounded-blast-radius hardening gaps, and minor doc inconsistencies. Grouped by theme:

### Code quality nits

- **Hand-rolled CUID** in `seed-launch-bots.mjs` (quality-first F4). 25 chars starting with `c` but body is base64url where real CUIDs are lowercase base36. Works today; would silently drop out if anyone adds a `cuid.isCuid()` filter. Fix: import the `cuid` package, or switch the script to `prisma.bot.create` + `prisma.botApiKey.create`.
- **Seed script duplicates `createBotForOwner` + `mintKey`** logic via raw pg (agent-native F3). The domain layer exists. Fix: have the seed script import `createBotForOwner` and follow with a `prisma.bot.update` for the tier.
- **Duplicate `BotRateTier` re-aliases** (sacred-schema F1, goldilocks F3). Three sources of truth: Prisma enum, `src/bots/index.ts` string union, admin route's `VALID_TIERS` array. Fix: re-export from `@/generated/prisma/enums` everywhere except `lib/rate-limit.ts` (which legitimately needs to stay edge-runtime-friendly).
- **`?since=<iso>` on `/events` is speculative** (goldilocks F2). No current bot uses it; the parser + Prisma branch can ship the day a real consumer needs it. Subsumed by P2.8's rewrite anyway.
- **`pnpm op bot:create-power` from the requirement was never implemented** (agent-native F2). Either ship it as a generic primitive or update the requirement to record the decision to compose `bot:create` + `admin:set-bot-tier`.

### Wire protocol nits

- **`viewers` 200 body has no `request_id`** + no outer try/catch (observability F3). An unexpected throw outside the rate-limit-helper branch surfaces as Next's generic 500 without `request_id`. Fix: outer try/catch returning structured `internal_error`.
- **Admin 404 auth-fail body omits `request_id`** (observability F3). Deliberate to not advertise the path — but means an operator who fat-fingered the token can't self-diagnose. Including a UUID doesn't leak path existence.
- **`viewers_read_failed` log omits `dependency: "upstash"`** (observability F2). The field is in `LogFields` for this exact purpose.
- **Cron-route error slugs distinguish "missing key" from "wrong CRON_SECRET"** (security F2). A guessing attacker who lands on the right CRON_SECRET observes a status change (404 → 500 = `missing_bot_key`) that confirms the guess. Fix: byte-identical 404 even after auth passes when keys are missing.

### Security/privacy hardening

- **Bot names unvalidated + publicly readable** via `/events` (security F3). Owners can stuff PII into a name field that's now anonymous-readable. Fix: add a character-class regex + length cap to `readNameBody`; document that bot names are public.
- **Launch-bot keys long-lived with no rotation cadence** (security F4). Plaintext in Vercel env, no detection signal for unexpected origin IPs. Fix: document a quarterly rotation cadence; add `pnpm admin:rotate-launch-bot-key <bot-name>` if rotation becomes routine.
- **Failed-admin-auth audit writes have no rate limit** (security F6). Unbounded INSERT against unauthenticated traffic. Inherited from M1's revoke-key endpoint; M2.5 amplifies by adding a second admin endpoint with the same pattern. Fix: add per-IP rate limit before the audit insert, or coalesce on `(source_ip, day)`.
- **Seed script audit rows record `source_ip: "local-script"`** (security F7). No operator identity captured. Fix: require `--actor <email>` flag, write to `payloadJson.invoked_by`.

### Operational nits

- **visitor-pulse Redis state key has no TTL or version namespace** (quality-first F5). `botplace:m25:visitor-pulse:last_blocks:sector-1` sits in Upstash forever if the bot is disabled. Fix: set a 7-day TTL, add `:v1:` namespace.
- **Admin set-tier endpoint skips the per-owner write rate limit** (quality-first F6). Every other credential-mutation route runs `checkOwnerWriteRateLimit`. The admin route is gated only by ADMIN_TOKEN. If the token ever leaks, an attacker can flip every bot to FREE faster than the operator notices.
- **Rollback section doesn't mention Upstash state cleanup** (release-and-rollout F3). `botplace:m25:visitor-pulse:last_blocks:sector-1` lingers after revert. Fix: add a one-line note in option (2) of the rollback section.
- **Risk R7 mitigation isn't asserted by a test** (release-and-rollout F3). The requirement promises that owner-facing routes hardcode FREE on creation. Add a unit test that `POST /api/v1/bots` with `rate_tier: "POWER"` in the body returns a FREE bot.
- **Sparkle log field is `sparkle_skipped: true`** but the probe doc says to grep `"skipped": true` (release-and-rollout F4). Inconsistent. Fix: rename log field to `skipped` to match.

### Cloud-coding plurality nits

- **Plaintext API keys land at `/tmp/m25-keys.json`** with manual cleanup (cloud-coding F4). A crashed cloud-agent leaves credentials on disk. Fix: pipe the seed script's JSON output directly to `vercel env add` per-bot, removing the intermediate file. Or store in 1Password via `op document create`.
- **`vercel env pull` + `source` is a parallel pattern to `pnpm op`** (cloud-coding F5). Two patterns, neither acknowledged as "the prod-secret pattern" vs "the dev-secret pattern." Fix: explainer paragraph in the probe + `secrets.md`.
- **Probe assumes `vercel`, `gh`, `jq`, `shred` on PATH without prerequisites block** (coding-agent-plurality F1). Fix: one-block "Prerequisites" header.
- **Probe uses `pnpm op admin:set-bot-tier` on line 148 but bare `pnpm m25:seed-launch-bots` on line 39** (coding-agent-plurality F2, markdown F2). Same workflow documented two ways. Fix: drop the `op` prefix on line 148 to match the rest of the M2.5 surface.
- **Prod secrets land at `/tmp` with no enforcement that the host is interactive** (coding-agent-plurality F3). Long-running cloud-agent runtimes share `/tmp`. Fix: prepend a "this probe must run on an ephemeral or operator-only host" callout, or use `mktemp -d` with `trap` cleanup.

### Documentation drift

- **M2 forward references to "M2.5 demo bots" never closed** (compound F5). M2 review + M2 requirement both describe M2.5 as "small, operator-side, possibly an ad-hoc script" — pre-rescope language. Fix: one-line pointer at the top of those two M2 docs to the M2.5 requirement.
- **`docs/api/v1.md` bare URL reference to admin endpoint doesn't link to admin/v1.md** (markdown F3). Fix: replace with explicit markdown link.
- **`docs/admin/v1.md` audit-events `action` example list omits `set_bot_rate_tier`** (markdown F4). Fix: add to the comma-separated list.
- **API doc changelog stops at M1** (markdown F5). Fix: add a 2026-05-12 entry naming M2 and M2.5 milestones.
- **Launch bots count their own egress** in visitor-pulse meter (core/agent-native F3, requirement decision #5). Deliberate; flag in docs for completeness.
- **`bot_name`-only attribution can collide across owners** (core/agent-native F2). `Bot.@@unique([ownerId, name])` is per-owner. Considered trade-off; revisit before M3 ships third-party-bot DX.

## What compounded forward (positive notes)

Worth recording explicitly. M2.5 reused M1/M2 patterns without reinvention:

- **Rate-limit framework** (`lib/rate-limit.ts`): the M1 `BucketConfig` shape parameterized cleanly; `checkPixelWriteRateLimit` extended with `tier?` defaulting to `"FREE"` so existing callers are back-compat. The M2 `PUBLIC_READ` floor was reused unchanged for the two new public endpoints.
- **Admin-auth pattern**: 404-on-failure (not 401), SHA-256-then-`timingSafeEqual`, failed-auth audit row — all reused verbatim from M1's revoke-key route.
- **Audit-trail conventions**: `set_bot_rate_tier` action with before/after `payloadJson` and `idempotent` flag mirrors M1's revoke-key shape. Seed script writes the same audit row a route would write.
- **Agent-native CLI parity**: every admin endpoint has an equivalent `pnpm admin:*` script in the same PR (`docs/admin/v1.md` line 51's codified rule).
- **Probe doc shape**: phased deployment with explicit pass criteria per phase + rollback recipe; `probes/README.md` updated to list the new probe (satisfying M2's P2.16).
- **Documentation discipline**: every code change has a corresponding doc update in the same PR; no drift at ship time on shape, only on the rename surfaced by P2.14.

### B-tier — partially landed, partially deferred to M3 (2026-05-13)

Landed in the B-tier cleanup PR (this commit set):

- **B2** — `BotRateTier` consolidated. Single source of truth at `@/generated/prisma/enums`; `src/bots/index.ts` re-exports it; admin set-tier route derives `VALID_TIERS` via `Object.values()`. `lib/rate-limit.ts` keeps its narrow string alias on purpose so it remains import-free of generated code.
- **B4** + **B5** — `/viewers` route gains outer try/catch returning structured `internal_error` with `request_id`; `viewers_read_failed` log gains `dependency: "upstash"`. `request_id` is now in the 200 body (documented in `docs/api/v1.md`).
- **B7** — `secrets.md` now has an explicit scope header (app owner/maintainer audience only; bot-author secrets are a separate M3 surface) plus a new "Two patterns for sourcing secrets into a shell" section disambiguating `pnpm op` (local dev with 1Password) from `vercel env pull` + `source` (production one-shot from operator workstation). The M2.5 probe doc links back to the prod-one-shot pattern.

Deferred to M3 with decision captured:

- **B3** — Bot identity drift across owners. **Decision: move to a handle + display-name model.** Bots get a stable, globally-unique `handle` (slug-shaped, used as the canonical identifier in API responses, logs, and across-owner attribution) plus a non-unique `displayName` (the existing `name` semantics — user-facing label that may collide across owners). The schema change, backfill of existing rows, API surface updates (`/events` exposes `handle`; ownership still scoped to owner), and validation rules (handle is strict, display name is permissive up to `MAX_NAME_LENGTH`) ride with M3 third-party bot DX. Not implemented now because the natural moment is when M3 first introduces non-operator bot authors.

Deferred indefinitely (no natural moment yet — re-open when one arises):

- **B1** — Seed script duplicates `createBotForOwner` + `mintKey` logic via raw pg. Works today; idempotent; three launch bots already seeded. Natural moment: adding a fourth launch bot, hitting a seed bug, or any change that touches the script's shape. This is also where the deprecated `cuid` dep (A5 carryover) gets retired, since the cleanest fix is to use the Prisma client and stop hand-generating CUIDs.
- **B6** — Plaintext keys at `/tmp/m25-keys.json` during seeding. The probe doc's Prerequisites + ephemeral-host callout + `shred -u` cleanup + the new jq-stdin recipe for `vercel env add` already cover the operational guidance. The structural fix (pipe seed output directly into `vercel env add`, no intermediate file) would gut the seed script's "JSON to stdout" shape and is bigger than it sounds. Defer until the seed flow changes.
- **B8** — Seed script audit `source_ip: "local-script"` has no actor identity. Single-actor world today. When a second operator first runs seed in anger, or M3 introduces shared-operator workflows, require `--actor <email>` and put it in `payloadJson.invoked_by`.

## Open Questions

- **Vercel project-env access scope.** P2.7's exploitability depends on whether project-env write is broader than DB-write for this team. If they're equivalent today, P2.7 downgrades to P3.
- **Will any near-term bot use `?since=` on `/events`?** If a follow-up reactive bot is already designed against this parameter, P2.8 becomes load-bearing immediately. If not, the lossy-pagination behavior can ship and the rewrite lands when the first lossless consumer appears.
